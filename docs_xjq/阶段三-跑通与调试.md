> 金牌技术导师视角 · 接续 `阶段二-微观核心剖析.md` · 这一篇让代码在你机器上「活起来」

## 一、最小可行性 Demo（MVP）：最少需要配置什么

这个项目依赖 4 个存储 + 1 个后端 + 1 个 worker + 1 个前端，听起来吓人，但「最小跑通」其实只要盯住**少数几个必填项**。

### 1.1 MVP 依赖清单（一个都不能少的）

| 必需项 | 为什么不能省 | 怎么起 |
|--------|-------------|--------|
| Docker（4 个存储） | PG/ES/Neo4j/Redis 缺一报错 | `docker compose up -d postgres elasticsearch neo4j redis` |
| 后端 api | 主程序 | `uv run python run.py` |
| Celery worker | 否则文档/记忆永远「处理中」 | `uv run celery -A app.celery_app.celery_app worker -l info -Q default,parse,memory,beat`（Win 加 `--pool=solo`） |
| 前端 web | 操作界面 | `npm run dev` |
| **1 个对话模型** | 不配无法问答 | 注册后在「模型配置」加 |
| **1 个 Embedding 模型** | 不配无法向量化/检索 | 同上 |

> Celery **beat** 是可选的（只管每日回顾/定时聚类），MVP 阶段可以不起，不影响问答和记忆萃取。

### 1.2 必改的配置项（只有这些是「必填」）

后端 `api/.env`（从 `api/.env.example` 复制）里，**只有两项必须改**，其余默认值都能直接用（因为存储地址默认指向 localhost，端口和 docker-compose 一致）：

```dotenv
# ① JWT 签名密钥 —— 随便一段长随机字符串
JWT_SECRET=随便敲一长串随机字符也行

# ② API Key 加密密钥（Fernet）—— 必须用命令生成，不能瞎填
FERNET_KEY=用下面命令生成的-44位base64
```

生成 `FERNET_KEY`（`config.py` 用它做 Fernet 对称加密存 API Key，格式不对会直接报错）：

```bash
uv run python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

> 根目录还有个 `.env`（给 docker-compose 用，配存储密码），`cp .env.example .env` 直接用默认即可，不用改。

### 1.3 数据库建表

```bash
cd api
uv run alembic upgrade head     # 跑 Alembic 迁移建表
```

> ES 索引和 Neo4j 图谱约束**不用手动建**——`main.py` 的 lifespan 钩子启动时自动建（见 `main.py:18-34`）。

### 1.4 最关键的「软配置」：LLM 模型（在前端 UI 里配，不在 .env）

这是新手最容易卡的地方：**模型配置不写在 .env，而是注册登录后在前端「设置 → 模型配置」里加，存进 PostgreSQL 的 `model_configs` 表**（API Key 用 Fernet 加密）。

为什么这么设计？因为这是**多用户**系统，每个用户用自己的 key、自己的模型。看 `model_config_model.py`：每条配置有 `type / provider / model_name / api_key / base_url / capability / is_default`。

**MVP 最少配两个**：

| 类型 type | 干什么 | 推荐 | 必填 |
|-----------|--------|------|:---:|
| `chat` | 对话/Agent 编排 | DeepSeek `deepseek-chat` | ✅ |
| `embedding` | 向量化（检索/记忆） | 智谱 `embedding-3`（维度 1024） | ✅ |
| `multimodal` | 看图问答 | 可选 | — |
| `rerank` | 检索重排 | 可选 | — |
| `websearch` | 联网搜索 | 可选 | — |

> ⚠️ **两个关键坑**：
> 1. **Embedding 维度必须是 1024**，要和 `config.py` 的 `EMBEDDING_DIMS=1024`、ES 索引维度一致，否则向量写入/检索报维度不匹配。智谱 `embedding-3` 正好支持指定 1024 维。
> 2. **强模型记得勾 `function_call` 能力**（`capability` 字段）。勾了走原生 Function Calling（路径 A），不勾走 ReAct（路径 B）。想验证两条路径，就靠这个开关切换——这是调试 Agent 推理路径的最简单办法。

每个模型加完点「测试连接」，通过后「设为默认」（代码靠 `is_default` 选默认，见 `chat_model.py:28`）。

### 1.5 一眼自检：服务是否就绪

```bash
curl http://localhost:8000/api/health     # 四个存储应全 "ok"
curl http://localhost:8000/api/hello      # 返回欢迎信息
```

`/api/health` 全 ok + 模型配好 + worker 在跑 = MVP 就绪。

### 1.6 最短「冒烟测试」路径

1. 注册 → 登录 → 配 chat + embedding 模型 → 设为默认。
2. 「记忆 → 我的画像」录一句：`我在腾讯做后端，养了只叫多多的小狗`，点「记住」。
3. 等 worker 日志打出「记忆萃取完成」（几秒~几十秒）。
4. 「对话」里问：`我养了什么宠物？`→ 观察 AI 自动调 `memory_search` 工具、流式答出「多多」。

跑通这条，说明 Agent 工具循环 + 记忆萃取 + 检索整条链路都通了。

## 二、关键断点设置：在哪打断点能看清「思考过程」

调试 Agent 的核心诉求是**看清「LLM 怎么决定调工具、工具返回了什么、怎么收敛到答案」**。下面按「一条问答请求的执行顺序」给出断点，按重要性标了 ⭐。

> 调试建议：用 VS Code 的 Python Debugger，配置 `run.py` 为入口（或用 `uv run python -m debugpy`）。**worker 里的代码（记忆萃取）要单独调试 Celery 进程**——最简单是临时把 `--pool=solo` 加上，让任务在单进程里同步跑，方便断点。

### 2.1 调试「问答 + 工具循环」（最核心，先看这条）

| ⭐ | 文件:行号 | 函数 | 在这里看什么 |
|----|----------|------|-------------|
| ⭐⭐⭐ | `services/chat_service.py:138` | `stream_chat` | 整条链路总入口。看加载的 `agent` 配置、`temperature`、`system_prompt` |
| ⭐⭐⭐ | `services/chat_service.py:146` | `_build_tools` 调用处 | 看本轮**启用了哪几个工具**（断点后 step into `_build_tools`） |
| ⭐⭐⭐ | `services/chat_service.py:173` | `supports_function_call(config)` | **决定走强模型还是弱模型路径**——看 config.capability |
| ⭐⭐⭐ | `core/agent/orchestrator.py:44` | `run_function_calling` | **强模型路径的心脏**：`tool_calls = ...`，看模型这一轮决定调哪些工具、传什么 query |
| ⭐⭐ | `core/agent/orchestrator.py:51-67` | 工具执行循环 | 看 `tool.ainvoke(args)` 的 `observation`（工具返回），以及 `messages` 怎么增长 |
| ⭐⭐ | `core/agent/orchestrator.py:95` | `run_react` | **弱模型路径**：`resp = await model.ainvoke(convo)`，看模型吐的原始 ReAct 文本 |
| ⭐⭐ | `core/agent/orchestrator.py:105-114` | ReAct 解析 | 看 `_ACTION_RE` 正则解析出的 `tool_name`/`query`，验证格式是否被正确解析 |

**调试强模型工具循环的标准动作**：
1. 在 `orchestrator.py:44` 打断点，问「我养了什么宠物」。
2. 第一次停下：看 `tool_calls`——应该有一个 `{"name": "memory_search", "args": {"query": "宠物"}}`。
3. 看 `observation`（`orchestrator.py:62` step over 后）：工具返回的记忆上下文文本。
4. 看 `messages` 列表：现在有了 `[..., AIMessage(要调工具), ToolMessage(工具结果)]`。
5. for 循环进入第二轮：模型这次 `tool_calls` 为空 → `yield final` → 这就是最终答案。

**亲眼走一遍这 5 步，你就彻底懂 Agent 了。**

### 2.2 调试「工具内部」（看检索到了什么）

| ⭐ | 文件:行号 | 看什么 |
|----|----------|--------|
| ⭐⭐ | `core/agent/tools.py:39` | 知识库工具：`hits = await hybrid_search(...)` 的命中结果 |
| ⭐⭐ | `core/agent/tools.py:49` | `citations.append(...)`——**引用是在这里收集的**，看溯源数据 |
| ⭐ | `core/agent/tools.py:77` | 记忆工具：`results = await search_memory(...)` |
| ⭐⭐ | `core/rag/search.py:115` | RAG 融合打分 `fused[cid] = 0.6*vec + 0.4*bm25`，看向量/BM25 各自命中 |
| ⭐ | `core/memory/retrieval/searcher.py:119` | 记忆检索融合排序，看 top_ids |

### 2.3 调试「记忆萃取」（看图谱怎么长出来的，在 worker 进程里）

| ⭐ | 文件:行号 | 看什么 |
|----|----------|--------|
| ⭐⭐⭐ | `core/memory/extraction/orchestrator.py:113` | 逐块循环入口，看分块结果 `chunks` |
| ⭐⭐⭐ | `core/memory/extraction/orchestrator.py:114` | `extract_statements` 返回——**原子陈述句**（带类型/时间） |
| ⭐⭐⭐ | `core/memory/extraction/orchestrator.py:119` | `extract_triplets_batch` 返回——**三元组+实体+事件** |
| ⭐⭐ | `core/memory/extraction/dedup.py:111` | `_judge_same` 的 LLM 去重判定结果——看它怎么决定两个实体是否同一 |
| ⭐⭐ | `core/memory/extraction/orchestrator.py:236` | `_persist` 落库前——看最终的 entities/relations/events 全貌 |
| ⭐ | `tasks/memory.py:54` | 任务入口 `run_extraction` 调用处，看入参 |

> 调 worker 的小技巧：直接在 `tasks/memory.py` 里临时写个 `if __name__ == "__main__": asyncio.run(_run("某个memory_id"))`，用普通 Python 调试器跑，比起 Celery 进程好断点得多。

### 2.4 用日志代替断点（更轻量）

不想打断点时，把日志调到 DEBUG（`api/.env`）：

```dotenv
LOG_LEVEL=DEBUG
DB_ECHO=true       # 打印所有 SQL（看 repository 层在查什么）
```

关键日志点已埋好：`orchestrator.py` 工具失败、`searcher.py` 召回降级、`orchestrator.py:251` 萃取完成统计（`记忆萃取完成: {...}`）都有日志。worker 终端直接能看到萃取产出了几个实体/关系/事件。

### 2.5 前端侧观察（最直观，零配置）

打开浏览器 DevTools → Network → 找 `/api/chat/stream` 请求 → 看 **EventStream** 标签，能直接看到后端吐的 SSE 事件流：

```
event: meta        → 会话信息
event: tool_call   → {"tool":"memory_search","query":"宠物"}   ← AI 决定调工具
event: token       → 逐字的回答
event: citation    → 引用列表
event: done        → 结束
```

**这是不打断点、最直观看清「Agent 调了哪些工具」的方式。**

## 三、本阶段小结

- **MVP 必改**：`api/.env` 的 `JWT_SECRET` + `FERNET_KEY`（命令生成）；前端配 `chat` + `embedding` 两个模型并设默认。
- **必起进程**：4 存储 + api + worker（+ 前端）。
- **看思考过程的黄金断点**：`orchestrator.py:44`（强模型 tool_calls）和 `orchestrator.py:95`（弱模型 ReAct 文本）。
- **零成本观察**：浏览器 Network 的 EventStream，或 worker 终端的萃取日志。
