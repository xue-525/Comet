## 一、实战魔改：给 Agent 加一个「查天气」自定义工具

我们以「查天气」为例，走一遍标准流程。**核心结论先说**：在这个项目里加工具，你**只需要动 2~3 个文件**，因为工具系统是开放设计的。

### 1.1 在哪个目录、按什么规范写

| 要做的事 | 在哪个文件 | 改动量 |
|---------|-----------|--------|
| ① 写工具的「干活逻辑」 | 新建 `core/agent/weather.py`（或复用现有） | 新增 |
| ② 把它包成 LangChain 工具 | `core/agent/tools.py` | 加一个 `build_weather_tool` |
| ③ 让问答时挂上这个工具 | `services/chat_service.py` 的 `_build_tools` | 加一行 |

规范要点（对照阶段二）：
- 工具一定是 `StructuredTool`，三件套：`name` / `description` / `args_schema`。
- `description` 写清楚「**什么场景该调我**」——LLM 全靠它决策。
- 需要的上下文（如 user_id、api_key）用**闭包**在 `build_xxx_tool` 工厂里注入，不要让 LLM 传。
- 工具函数返回**字符串**（给 LLM 看的自然语言），不是结构化对象。

### 1.2 标准模板

**Step ①：写干活逻辑** `api/app/core/agent/weather.py`（新建）

```python
"""天气查询：调用公开天气 API，返回给 Agent 用的天气文本。"""
import httpx

from app.core.logging import get_logger

logger = get_logger(__name__)


async def query_weather(city: str) -> str:
    """查某城市天气，返回拼好的中文描述。失败抛异常由工具层兜底。"""
    # 示例用 wttr.in（无需 key）；真实场景换成和风/OpenWeather 并从配置取 key
    async with httpx.AsyncClient(timeout=15) as client:
        resp = await client.get(f"https://wttr.in/{city}?format=j1")
        resp.raise_for_status()
        data = resp.json()
    cur = data["current_condition"][0]
    desc = cur["lang_zh"][0]["value"] if cur.get("lang_zh") else cur["weatherDesc"][0]["value"]
    return f"{city} 当前天气：{desc}，气温 {cur['temp_C']}°C，体感 {cur['FeelsLikeC']}°C，湿度 {cur['humidity']}%。"


__all__ = ["query_weather"]
```

**Step ②：包成工具** 在 `api/app/core/agent/tools.py` 里加（对照现有 `build_web_search_tool`）：

```python
# 1) 顶部加工具名常量
TOOL_WEATHER = "weather_query"


# 2) 入参 schema（和 _QueryInput 同理，但语义是城市）
class _CityInput(BaseModel):
    city: str = Field(..., description="要查询天气的城市名，如 北京、上海")


# 3) 工厂函数
def build_weather_tool() -> StructuredTool:
    """天气查询工具：调用天气 API。"""

    async def _run(city: str) -> str:
        from app.core.agent.weather import query_weather
        try:
            return await query_weather(city)
        except Exception as e:
            logger.warning("天气查询失败: %s", e)
            return f"天气查询失败：{e}"

    return StructuredTool.from_function(
        coroutine=_run,
        name=TOOL_WEATHER,
        description="查询指定城市的实时天气。当用户问某地天气、气温、是否下雨时使用。",
        args_schema=_CityInput,
    )
```

别忘了把 `TOOL_WEATHER` / `build_weather_tool` 加进文件底部的 `__all__`。

**Step ③：挂到问答链路** 在 `api/app/services/chat_service.py` 的 `_build_tools`（约 `chat_service.py:92`）里加一行：

```python
from app.core.agent.tools import build_weather_tool   # 顶部 import 处补上

async def _build_tools(self, user_id, agent, body, citations):
    ...
    tools: list = []
    ...（现有三个工具）...
    tools.append(build_weather_tool())     # ← 加这一行（天气工具无需上下文，最简单）
    return tools
```

**完成。** 重启后端，问「北京今天天气怎么样」，强模型会自动调 `weather_query` 工具。整个过程**没动 orchestrator、没动前端核心逻辑**——这就是「工具系统开放、编排逻辑闭合」的好处。

### 1.3 进阶：让它像现有工具一样「可配开关 + 注入上下文」

如果你的工具需要 user_id、需要从用户配置取 api_key，模仿 `build_knowledge_tool`：

```python
def build_weather_tool(session, user_id):       # 工厂接收上下文
    async def _run(city: str) -> str:
        # 闭包里能用 session/user_id 去查用户配置、做隔离
        api_key = await _get_user_weather_key(session, user_id)
        ...
    return StructuredTool.from_function(coroutine=_run, ...)
```

如果想做成「可开关」（像联网那样默认关），就：
1. 在 `AgentConfig` 模型加字段 `enable_weather`（需写 Alembic 迁移）；
2. 在 `_build_tools` 里按 `enabled(body.enable_weather, agent.enable_weather)` 判断。

### 1.5 「读本地文件」工具——再来一个例子巩固

intro.md 还提到「读本地文件」。要点完全一样，只是 `_run` 换成读文件，**但务必注意安全**（这是真实开发会被审查的点）：

```python
class _PathInput(BaseModel):
    path: str = Field(..., description="要读取的文件相对路径")

def build_read_file_tool(base_dir: str) -> StructuredTool:
    async def _run(path: str) -> str:
        from pathlib import Path
        # ⚠️ 安全：强制限制在 base_dir 内，防目录穿越
        target = (Path(base_dir) / path).resolve()
        if not str(target).startswith(str(Path(base_dir).resolve())):
            return "拒绝访问：路径越界"
        if not target.is_file():
            return f"文件不存在：{path}"
        return target.read_text(encoding="utf-8")[:4000]   # 截断防超长
    return StructuredTool.from_function(
        coroutine=_run, name="read_file",
        description="读取指定本地文件的内容。当需要查看某个文件时使用。",
        args_schema=_PathInput,
    )
```

> 💡 工具开发三条铁律：**① 限制范围（防越权/穿越）；② 截断输出（防 token 爆炸）；③ 失败返回字符串而非抛异常（让 Agent 能继续）**。这个项目现有工具全部遵守，照抄即可。

## 二、落地建议：从零写一个轻量级 Agent，先写哪几个类

如果你想脱离这个项目、自己从零撸一个最小 Agent，**别一上来就上 LangChain 全家桶**。参照本项目的「克制用法」，按下面顺序写 5 个核心类/模块就够了，**由内到外**：

### 2.1 五个核心类（按编写顺序）

```
① LLMClient        ——  和大模型说话的嘴
② Tool             ——  Agent 能用的手
③ AgentExecutor    ——  把嘴和手串起来的工具循环（Agent 的大脑/心脏）
④ Memory           ——  记住上下文（先做短期，再考虑长期）
⑤ ChatService/API  ——  对外的入口
```

#### ① `LLMClient`（先写这个，最底层）
封装「给一组 messages，返回模型回复」。参考本项目 `core/llm/client.py`——它就是一个 httpx 包了 OpenAI 兼容接口的薄类。**最小实现只需一个方法**：

```python
class LLMClient:
    def __init__(self, base_url, api_key, model): ...
    async def chat(self, messages: list[dict], tools: list | None = None) -> dict:
        # POST /chat/completions，把 tools 的 schema 一起传，返回 message（可能含 tool_calls）
```

关键：把 `tools` 的 JSON Schema 一起 POST 给模型，这是 function calling 的基础。

#### ② `Tool`（定义"手"）
最小抽象：一个 `name` + `description` + `参数 schema` + 一个 `run(args)` 函数。本项目直接用 LangChain `StructuredTool`，你自己写可以更简单：

```python
@dataclass
class Tool:
    name: str
    description: str
    parameters: dict            # JSON Schema
    func: Callable              # 实际执行

    def to_openai_schema(self) -> dict:   # 转成 OpenAI function 格式
        return {"type": "function", "function": {
            "name": self.name, "description": self.description,
            "parameters": self.parameters}}
```

#### ③ `AgentExecutor`（最核心——工具循环）
这是 Agent 的灵魂，**直接照搬本项目 `orchestrator.py:run_function_calling` 的 for 循环**：

```python
class AgentExecutor:
    def __init__(self, llm: LLMClient, tools: list[Tool], max_iter=5):
        self.llm = llm
        self.tool_map = {t.name: t for t in tools}
        self.schemas = [t.to_openai_schema() for t in tools]
        self.max_iter = max_iter

    async def run(self, messages: list[dict]) -> str:
        for _ in range(self.max_iter):
            resp = await self.llm.chat(messages, tools=self.schemas)
            tool_calls = resp.get("tool_calls")
            if not tool_calls:
                return resp["content"]                  # 没调工具 → 最终答案
            messages.append(resp)                        # AI 的"我要调工具"
            for tc in tool_calls:                        # 执行每个工具
                tool = self.tool_map[tc["name"]]
                result = await tool.func(**tc["args"])
                messages.append({"role": "tool",
                                 "tool_call_id": tc["id"],
                                 "content": str(result)})  # 结果回灌
        return "（超过最大轮数）"
```

**这 15 行就是一个能用的 Agent。** 本项目只是在它外面包了流式、双路径、引用收集等工程细节。

#### ④ `Memory`（记上下文）
- **第一版只做短期**：一个 list 存历史消息，截断到最近 N 轮。照搬 `chat_service.py` 的 `MAX_HISTORY_TURNS` 思路即可。
- **要长期记忆再加**：才需要本项目那套「萃取→图谱→检索」的重型方案。**不要一开始就上图数据库**。

#### ⑤ `ChatService` / API 入口
把上面四个串起来，对外暴露一个 `chat(user_input) -> answer`。要流式就用 SSE（FastAPI `StreamingResponse`），参考 `chat_controller.py:81`。

### 2.2 写作顺序与心法

```
第 1 天：LLMClient + 一个最简单的 chat()           → 能和模型对话
第 2 天：Tool + AgentExecutor 工具循环              → 能调工具（这是 Agent 的"成人礼"）
第 3 天：加 2~3 个真实工具（搜索/计算/查库）         → 有用了
第 4 天：加短期 Memory + 流式输出                    → 体验好了
之后：  按需加 长期记忆 / 多模态 / 弱模型降级 / 多租户
```

**心法三条**：
1. **工具循环是 Agent 的本质**，先把 `AgentExecutor.run` 那 15 行写对，其他都是锦上添花。
2. **LangChain 用最底层的部分就好**（消息类型、StructuredTool、bind_tools），别用 `AgentExecutor` 黑盒——本项目就是这么做的，让核心逻辑掌握在自己手里。
3. **先单用户、先短期记忆、先一个强模型**，跑通后再逐步加 多租户/长期记忆/弱模型降级，**别一开始就追求工业级**。

### 2.3 本项目可直接「抄」的设计

学完这个项目，下面这些设计你可以直接搬进自己的 Agent：

| 设计 | 出处 | 价值 |
|------|------|------|
| 工具循环 for-loop + max_iter 防死循环 | `orchestrator.py:35` | Agent 核心 |
| 闭包注入工具上下文（user_id/citations） | `tools.py:36` | 干净地隔离「LLM 控制的参数」和「环境」 |
| 统一事件流解耦编排与输出 | `orchestrator.py` | 换推理策略不动上层 |
| 强/弱模型双路径降级 | `chat_service.py:173` | 兼容各档模型 |
| 工具失败返回字符串而非抛异常 | 所有 `build_xxx_tool` | Agent 能继续而不崩 |
| 向量+全文混合召回+加权融合 | `search.py` / `searcher.py` | RAG/记忆检索通用配方 |
| 分层 + 文件名后缀约定 | 整个 `api/app` | 大项目可维护性 |
