## 一、🛠️ Tools/Actions：工具怎么定义、怎么被触发

### 1.1 工具是什么——LangChain `StructuredTool`

看 `core/agent/tools.py`。每个工具就是一个 `StructuredTool`，由三部分构成：

```python
# tools.py:18  ——  工具的入参 schema（告诉 LLM 该传什么）
class _QueryInput(BaseModel):
    query: str = Field(..., description="检索的问题或关键词")

# tools.py:57  ——  把一个 async 函数包成工具
return StructuredTool.from_function(
    coroutine=_run,                         # 工具真正干活的协程
    name=TOOL_KNOWLEDGE,                     # "knowledge_search"
    description="从用户的个人知识库（文档、图片）中检索相关内容……",  # ← LLM 靠它决定要不要调
    args_schema=_QueryInput,                 # 入参 schema
)
```

**三个关键字段，决定了 LLM 怎么用工具**：
- `name`：工具唯一标识（前端按它映射图标，见 `TOOL_KNOWLEDGE/MEMORY/WEB` 三个常量）。
- `description`：**最重要**。LLM 完全靠这段中文描述判断「这个问题该不该调这个工具」。写工具时，description 就是你的「提示词工程」主战场。
- `args_schema`：用 Pydantic 描述入参，LangChain 会把它转成 OpenAI function calling 的 JSON Schema。

项目里有**三个工具**（`tools.py`）：

| 工具 | name | 内部调用 | description 触发条件 |
|------|------|---------|---------------------|
| 知识库 | `knowledge_search` | `core/rag/search.hybrid_search` | 问题涉及用户上传的资料/文档/笔记 |
| 记忆 | `memory_search` | `core/memory/retrieval/searcher.search_memory` | 问题涉及「我」的个人信息 |
| 联网 | `web_search` | `core/agent/web_search.web_search` | 需要实时信息/最新新闻 |

### 1.2 工厂模式 + 闭包注入上下文（重要技巧）

每个工具不是全局单例，而是**每次问答现场构建**（`build_xxx_tool` 工厂函数）。看知识库工具（`tools.py:28-62`）：

```python
def build_knowledge_tool(session, user_id, citations, embed_client_holder):
    async def _run(query: str) -> str:           # ← 闭包：捕获了外层的 session/user_id/citations
        hits = await hybrid_search(session, user_id, query, top_k=5)
        ...
        citations.append({...})                   # ← 工具执行时，直接把引用写进外部列表
        return "检索到以下知识库内容：\n\n" + "\n\n".join(parts)
    return StructuredTool.from_function(coroutine=_run, ...)
```

**为什么这么设计？** 因为 LLM 调工具时只会传 `query` 这一个参数，它**不知道也不该知道** `user_id`、数据库 session 这些上下文。用闭包在构建期把这些「环境」注入进去，工具被触发时就能：
1. 按 `user_id` 做多租户隔离检索；
2. 把检索到的来源直接 `append` 进共享的 `citations` 列表——**引用溯源就是这么收集的**（编排结束后 `ChatService` 从这个列表读出来发 `citation` 事件）。

> 💡 学习点：**「工具的入参 = LLM 能控制的；工具的上下文 = 闭包注入的」**。这是给工具注入「当前用户/会话」最干净的写法，比用全局变量或线程局部变量优雅得多。

`embed_client_holder` 是个 `dict`，作用是**跨工具共享 embedding 客户端**（懒加载一次，知识库和记忆工具都能用，避免重复构建），见 `tools.py:70-76`。

### 1.3 工具按开关动态组装

回到 `chat_service.py:92` 的 `_build_tools`：工具不是固定三个，而是按「Agent 默认开关 + 本轮请求覆盖」决定：

```python

    return default if override is None else override   # 本轮没指定就用 Agent 默认值

a_know = agent.enable_knowledge if agent else True     # 知识库默认开
```

这对应 `AgentConfig` 表的三个 `Boolean` 字段（`agent_config_model.py`）。**联网默认关**是个产品考量——联网要花钱、要配 key、还可能引入噪声，所以默认关、按需开。

### 1.4 工具怎么被「触发执行」——这才是 Function Calling 的核心

工具定义好了，**谁来决定调哪个、什么时候调、传什么 query？** 答案是 LLM，机制在 `orchestrator.py`。这部分和「Planning」高度重叠，所以放到第二节一起讲。

## 二、⚙️ Planning/Reasoning：双路径工具循环

这是整个项目**最该精读的 130 行**：`core/agent/orchestrator.py`。它实现了「方案 B」——**根据模型能力走两条不同的推理路径**，但对外暴露统一事件流。

### 2.1 为什么要两条路径？

```python
# chat_service.py:173
elif supports_function_call(config):
    run_function_calling(...)   # 强模型：原生 Function Calling
else:
    run_react(...)              # 弱模型：提示词模拟 ReAct
```

判定依据是 `supports_function_call`（`chat_model.py:54`）——看模型配置的 `capability` 数组里有没有 `"function_call"` 标记（用户在「模型配置」里手动勾的）：

- **强模型**（如 GPT-4o、DeepSeek-Chat 支持原生工具调用）：用 OpenAI 的 function calling 协议，模型直接吐结构化的 `tool_calls`，**准确、稳定**。
- **弱模型**（不支持/不稳定的小模型）：退化用 **ReAct 提示词**，让模型按 `Thought/Action/Action Input` 格式输出文本，代码用正则解析。**兼容性好，但靠格式约定，没那么稳**。

> 💡 这就是工业级 Agent 的「优雅降级」：能用原生就用原生，不能用就退到提示词方案，**保证各档模型都能跑**。

### 2.2 路径 A：原生 Function Calling 循环（`run_function_calling`, orchestrator.py:25）

这是 Agent 工具循环的「教科书实现」，精读：

```python
async def run_function_calling(model, tools, messages):
    tool_map = {t.name: t for t in tools}
    model_with_tools = model.bind_tools(tools) if tools else model   # ① 把工具 schema 绑给模型
    full_text = ""

    for _ in range(MAX_TOOL_ITERATIONS):          # ② 最多循环 5 轮，防死循环
        gathered = None
        async for chunk in model_with_tools.astream(messages):    # ③ 流式拿模型输出
            if chunk.content:
                full_text += chunk.content
                yield {"type": "token", "text": chunk.content}     #   边出边吐 token
            gathered = chunk if gathered is None else gathered + chunk   # 累积分片

        tool_calls = getattr(gathered, "tool_calls", None) or []
        if not tool_calls:                         # ④ 模型这轮没要求调工具 → 收敛，结束
            yield {"type": "final", "text": full_text}
            return

        messages.append(gathered)                  # ⑤ 把模型的「我要调工具」消息加回历史
        for tc in tool_calls:                      # ⑥ 逐个执行工具
            name, args = tc["name"], tc["args"]
            yield {"type": "tool_call", "tool": name, "query": args.get("query","")}
            tool = tool_map.get(name)
            observation = await tool.ainvoke(args)  #   真正执行工具
            messages.append(ToolMessage(content=str(observation), tool_call_id=tc["id"]))  # ⑦ 结果回灌

    # 5 轮还没收敛，用现有内容兜底
    yield {"type": "final", "text": full_text or "（未能生成回答）"}
```

**把这个循环刻进脑子**，它就是 Agent 的本质：

```
┌─────────────────────────────────────────────────────┐
│  ① bind_tools 把工具描述塞给模型                       │
│  ② 问模型 → 模型要么直接回答，要么说"我要调 X 工具"    │
│  ③ 若要调工具 → 执行 → 把结果(ToolMessage)拼回对话      │
│  ④ 带着工具结果再问模型 → 回到 ②                       │
│  ⑤ 直到模型不再要调工具 → 那就是最终答案               │
│     （最多转 MAX_TOOL_ITERATIONS=5 圈）                │
└─────────────────────────────────────────────────────┘
```

`messages` 这个列表在循环里**不断增长**：原始问题 → AI「我要调工具」→ ToolMessage 工具结果 → AI 再思考……这就是 Agent 的「工作记忆 / scratchpad」。

### 2.3 路径 B：ReAct 提示词循环（`run_react`, orchestrator.py:78）

弱模型没有原生 tool_calls，怎么办？用**提示词逼它按格式输出，再用正则解析**。

先看提示词模板 `core/agent/prompts/react.jinja2`：

```
请按以下格式逐步推理（ReAct）：
Thought: 你的思考，判断是否需要调用工具、调用哪个
Action: 工具名（必须是上面列出的工具名之一）
Action Input: 传给工具的查询关键词（一行纯文本）
... 调用工具后我会把结果作为 Observation 返回给你 ...
当你已经有足够信息回答时：
Final Answer: 给用户的最终回答
```

再看解析逻辑（`orchestrator.py:73-129`），关键是三个正则：

```python
_ACTION_RE       = re.compile(r"Action\s*:\s*(.+)")
_ACTION_INPUT_RE = re.compile(r"Action\s*Input\s*:\s*(.+)")
_FINAL_RE        = re.compile(r"Final\s*Answer\s*:\s*(.*)", re.DOTALL)
```

循环逻辑（伪代码）：

```
for 最多 5 轮:
    resp = model.ainvoke(convo)            # 注意：这里是非流式 ainvoke
    if 命中 "Final Answer:":               # → 解析出最终答案，结束
        yield final; return
    if 命中 "Action:" / "Action Input:":   # → 解析工具名和 query
        observation = 执行该工具
        convo.append(AIMessage(模型上轮输出))
        convo.append(HumanMessage(f"Observation: {observation}"))   # 把观察结果拼回去
    else:                                  # 既没 Action 也没 Final → 整段当回答兜底
        yield text; return
```

**ReAct = Reasoning + Acting**：模型先 `Thought`（推理）再 `Action`（行动），拿到 `Observation` 后再推理……和路径 A 是**同一个循环骨架**，区别只是「模型表达调用意图的方式」从结构化 `tool_calls` 变成了纯文本格式约定。

> 提示词策略对照（intro.md 问到的 ReAct/Plan-and-Solve/CoT）：
> - 本项目**显式用了 ReAct**（弱模型路径），见 `react.jinja2`。
> - 强模型路径是**原生 Function Calling**，本质也是「推理→行动→观察」的 ReAct 思想，只是由 OpenAI 协议承载，不需要手写格式。
> - **没有**用 Plan-and-Solve（先列完整计划再执行）或显式 CoT 模板——它走的是「边想边做」的交错式（interleaved）ReAct，更适合工具型 Agent。

### 2.4 统一事件流：双路径如何解耦

两条路径**产出完全一样的事件 dict**（这是设计精髓）：

```python
{"type": "tool_call", "tool": ..., "query": ...}
{"type": "token",     "text": ...}
{"type": "final",     "text": ...}
```

于是上层 `ChatService`（`chat_service.py:180-195`）用同一套代码消费，再翻译成 SSE：
- `token` → SSE `token` 事件（前端逐字显示）
- `tool_call` → SSE `tool_call` 事件（前端画「🔍正在检索知识库」标记）
- `final` 不单独发（token 已累积成完整文本）

**「双实现、单接口」**——换底层推理策略，上层完全不用改。这是策略模式（Strategy Pattern）在 Agent 编排里的漂亮应用。

## 三、🧠 Memory：长短期记忆与图谱

### 3.1 短期记忆（上下文记忆）

最简单的一层，在 `chat_service.py`：

```python
MAX_HISTORY_TURNS = 10                              # chat_service.py:47

async def _history_messages(self, conv_id):         # chat_service.py:71
    history = await self.msg_repo.recent_history(conv_id, MAX_HISTORY_TURNS)
    # 把 DB 里的历史消息转成 LangChain 的 HumanMessage / AIMessage
```

就是把**当前对话最近 10 轮**消息从 PostgreSQL 取出，转成 LangChain 消息，拼在 system prompt 之后、当前问题之前。这是给 LLM 的「上下文窗口记忆」，简单但有效。**注意它只在单个 conversation 内有效，换个会话就清零**。

### 3.2 长期记忆：写入侧——四层溯源图谱萃取

这是项目的灵魂。入口 `core/memory/extraction/orchestrator.py` 的 `run_extraction`（阶段一已概览，这里抠细节）。

**四层溯源图谱结构**（`graph_models.py:1-16`）：

```
Dialogue（来源：一段对话/一条主动记住的文本）
   └─HAS_CHUNK→ Chunk（切分的片段）
        └─HAS_STATEMENT→ Statement（原子陈述句，带类型+时间）
              └─MENTIONS→ Entity（萃取出的实体）

语义层叠加：
   Entity ─RELATION{predicate}→ Entity   （三元组关系）
   Event  ─INVOLVES→ Entity              （事件，带 event_time → 时间线）
   Entity ─IN_COMMUNITY→ Community        （社区聚类）
```

**为什么要四层？** 为了**溯源**——任何一个记忆事实，都能回答「这是从哪段对话、哪个片段、哪句话来的」。这是工业级记忆系统区别于「简单存原文」的关键：**可追溯、可验证、可解释**。

萃取流水线（`orchestrator.py:67-252`）的关键步骤：

**① 分块** → `chunker.split_chunks(text)`，长文本切成小块。

**② 逐块「陈述抽取」**（`statement_extractor.py`）：调 LLM 把文本切成**原子陈述句**，每句带两个属性（`graph_models.py:43-51`）：
- `stmt_type`：FACT（事实）/ OPINION（观点）/ PREDICTION（预测）/ SUGGESTION（建议）
- `temporal_type`：STATIC（静态）/ DYNAMIC（动态）/ ATEMPORAL（无时间）
- 还会标 `has_unsolved_reference`（指代未解析，如"他""那个"），这种句子下一步直接跳过。

**③ 逐句「三元组萃取」**（`triplet_extractor.py`）：调 LLM 抽 `(主语, 谓词, 宾语)` 三元组 + 实体 + 事件，**受控词表约束**（`ontology.py` 的 13 类实体 + 13 类谓词）。用 `extract_triplets_batch` 并发跑（`Semaphore(4)` 限并发防限流），`temperature=0.1`（要稳定，不要发散）。

**④ 实体名向量化** → `embedder.embed_texts`，给每个实体的 name 算 embedding，供去重和检索。

**⑤⑥ 两层去重**（`dedup.py`，**这是萃取最精妙的部分**）：
- **第一层 批内去重**（`dedup_within_batch`）：本次萃取出的实体之间消歧。先规则初筛（同名同类型直接合并、名称相似度 `difflib` ≥ 0.80、向量余弦相似、包含关系），候选对再交给 LLM 判定（`dedup_entity.jinja2`，置信度 ≥ 0.80 才合并）。
- **第二层 图侧融合**（`merge_with_graph`）：去重后的实体再和 Neo4j 里**已有的同类型实体**比对，命中就复用已有节点 id。

> 💡 为什么去重这么重要？因为「我」「腾讯」这种实体会在无数次对话里被反复提及，如果每次都新建节点，图谱就炸了。两层去重保证**同一实体跨多次萃取只有一个图节点**。注意 `merge_with_graph:144` 有个细节——**同名同类型直接复用，不问 LLM**，就是为了避免 LLM 的非确定性把「用户」反复判成不同实体。

**⑦⑧⑨ id 重定向**：去重会让实体 id 变化，所以所有 mention 边、relation 边、event 的参与者都要用 `resolve()`（`orchestrator.py:173`）把旧 id 映射到最终落库 id。这是去重后保证图连通性的「收尾工作」。

**⑩ 原子落库** → `MemoryGraphRepository.save_graph`，单事务写 Neo4j（按业务键 MERGE 幂等）。

**⑪ 增量社区聚类**（`label_propagation.py`）：新实体用标签传播算法归入社区，失败不影响主流程。

**整条流水线跑在 Celery worker 里**（`tasks/memory.py`），异步、不阻塞问答。触发点有两个：
- 「主动记住」：用户在记忆页手动录入 → `memory_service` 落库 → 派发任务。
- 「对话自动萃取」：每次问答后 `chat_service._dispatch_memory` 把用户那句话存成 `source=auto` 并派发。

### 3.3 长期记忆：读取侧——混合召回 + 一跳关系遍历

入口 `core/memory/retrieval/searcher.py` 的 `search_memory`（被 `memory_search` 工具调用）。流程：

```
query
 → ① 向量召回：query 向量化 → Neo4j 向量索引找相似实体（recall_size=20）
 → ② 全文召回：Neo4j 全文索引（cjk 中文分词）找名称匹配实体
 → ③ 归一化 + 加权融合（向量 0.6 + 全文 0.4）→ 取 top_k
 → ④ 一跳邻居遍历：对命中实体取其 RELATION 关系
 → 拼成「实体 + 关联事实」上下文
```

最后 `format_memory_context`（`searcher.py:150`）把结果拼成给 LLM 的文本：

```
- 多多（宠物）：用户养的小狗
    · 多多 属于 用户
- 周杰伦演唱会（事件）：...
    · 用户 参加 周杰伦演唱会
```

> 注意 `min_vector_score` 参数（`searcher.py:78`）：全局搜索时启用「绝对余弦门控」，只保留相似度 ≥ 0.45 的，宁缺毋滥；而 Agent 工具调用时不传它，走加权融合（召回优先）。**同一个检索函数，两种精度策略**。

### 3.4 顺带：知识库检索（RAG）也是同一套混合检索思想

`core/rag/search.py` 的 `hybrid_search`（被 `knowledge_search` 工具调用）和记忆检索是**同构**的：向量召回（ES knn）+ BM25 召回 → 归一化加权融合（0.6/0.4）→ 可选 rerank → 返回。两个亮点：
- **small-to-big**（`_resolve_parent_content`, `search.py:178`）：检索命中小的「子块」（精准），但返回大的「父块」（上下文完整）。
- **可选 rerank**：用户配了 rerank 模型才走，没配就用融合排序兜底。

> 记住这个范式：**「向量召回保语义 + BM25/全文召回保关键词 + 归一化加权融合 + 可选 rerank 精排」**，是 RAG 和图谱检索通用的工业级配方。
