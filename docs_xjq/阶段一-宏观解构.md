## 0. 先用一句话定位这个项目

> Comet 是一个**多用户的「个人 AI 知识库 + 记忆助手」**：把你的文档/图片/网页沉淀成可语义检索的知识库，从对话里自动萃取「记忆」构建你的专属知识图谱，再让一个 **LLM Agent 自主编排「知识库 / 记忆 / 联网」三类工具**来回答问题。

对学 Agent 的人来说，这个项目的「教科书价值」在于：它不是一个玩具 demo，而是一个**工业级分层架构 + 真实工具编排 + 异步任务流水线**的完整样本。你能在这里看到：
- Agent「工具循环」的两条真实路径（强模型原生 Function Calling vs 弱模型 ReAct 降级）；
- RAG 检索、记忆图谱萃取这些「重活」是怎么被拆进异步队列的；
- 一个 LLM 应用在生产里要操心的事（多租户、密钥加密、流式输出、四种存储各司其职）。

## 1. 鸟瞰架构：设计模式与文件夹角色

### 1.1 顶层：经典「前后端分离 + 存储外置」三件套

```
浏览器  ──▶  web (React/Vite, 5173)  ──/api 代理──▶  api (FastAPI, 8000)
                                                        │
                  ┌──────────────┬───────────┬─────────┴────────┐
                  ▼              ▼           ▼                  ▼
             PostgreSQL    Elasticsearch   Neo4j             Redis
             业务数据      向量+全文检索    记忆图谱       缓存 + 消息队列
                                                              │ broker
                                              ┌───────────────┴───────────────┐
                                              ▼                               ▼
                                       celery worker                    celery beat
                                       解析/萃取/聚类                   定时回顾/聚类
```

记住一个关键设计取舍：**四种存储各管一摊，没有用「一个数据库硬扛所有场景」**。这是工业级 Agent 的典型思路——
- **PostgreSQL**：结构化业务数据（用户、对话、消息、模型配置、记忆原文……）。
- **Elasticsearch**：知识库的「向量 + BM25 全文 + IK 中文分词」混合检索。
- **Neo4j**：记忆图谱（实体—关系—事件的四层溯源图，天然适合图查询）。
- **Redis**：既当缓存，又当 Celery 的消息中间件（broker）。

> 💡 学习提示：当你以后自己设计 Agent，「**检索用向量库、关系用图库、业务用关系库、异步用消息队列**」这套组合拳是值得抄的范式。

### 1.2 后端：严格单向的四层架构（DDD-lite 分层模式）

后端是这个项目的主体，采用**严格单向分层**，这是它最重要的「设计模式」：

```
controllers/   路由层（HTTP 入口，只做参数校验 + 调 service + 包装响应）
     │  ↓ 只能向下调用，不能反向
services/      业务逻辑层（编排：决定先做什么后做什么）
     │  ↓
repositories/  数据访问层（CRUD，屏蔽底层存储细节；含 neo4j/ 子目录）
     │  ↓
models/        SQLAlchemy ORM 模型 + db/ 连接
```

> 规矩（见 README「开发约定」）：`controller → service → repository → model/db`，**只能向下，不能跨层、不能反向**。文件名后缀强约定：`xxx_controller.py / xxx_service.py / xxx_repository.py / xxx_model.py / xxx_schema.py`。看到文件名就知道它在哪一层、干什么——这对新人读代码极其友好。

各文件夹角色一览（`api/app/` 下）：

| 目录 | 角色 | 类比 |
|------|------|------|
| `controllers/` | 路由层，HTTP 入口 | 餐厅的「点餐台」 |
| `services/` | 业务编排，决定流程 | 「后厨主厨」，统筹谁先做 |
| `repositories/` | 数据存取，屏蔽存储细节 | 「仓库管理员」 |
| `models/` | SQLAlchemy ORM 表结构 | 「货物清单格式」 |
| `schemas/` | Pydantic 请求/响应模型 | 「点餐单/出餐单的格式」 |
| **`core/`** | **横切基础设施（项目的灵魂）** | 「中央厨房的核心设备」 |
| `tasks/` | Celery 异步任务 | 「需要慢炖的菜，交给另一个炉子」 |
| `db/` | 四种存储的连接管理 | 「水电气网四条管线」 |
| `config.py` | 配置（pydantic-settings 读 .env） | 「总配电箱」 |
| `main.py` | FastAPI 入口（启动/关闭钩子） | 「开门营业的总开关」 |
| `celery_app.py` | Celery 多队列配置 | 「慢炖炉的排班表」 |

### 1.3 `core/` —— Agent 的真正「大脑与器官」都在这

这是你学 Agent 最该盯的目录。它把「跨业务复用的能力」沉淀成独立模块：

| `core/` 子模块 | 职责 | 阶段二会重点剖析 |
|---------------|------|:---:|
| `core/agent/` | **Agent 工具编排**：工具定义、工具循环（Function Calling / ReAct 双路径）、联网搜索 | ⚙️🛠️ |
| `core/memory/` | **记忆系统**：预处理/萃取/检索/聚类 + 提示词模板（写入 Neo4j 四层图谱） | 🧠 |
| `core/rag/` | **知识库检索**：解析/分块/索引/混合检索/图片描述 | 🧠 |
| `core/llm/` | **LLM 客户端与工厂**：按用户模型配置实例化 ChatOpenAI、能力判定 | — |
| `core/storage/` | 文件存储抽象（本地 / OSS 可切换） | — |

> 一句话记住分工：**`core/agent` 是「指挥中枢」，`core/rag` 和 `core/memory` 是它能调用的「两只手」，`core/llm` 是它的「发声器官」。**

### 1.4 前端（`web/src/`）

标准 React 单页应用，与本文 Agent 主题关系不大，知道角色即可：
- `api/`（请求封装，client + 各模块）、`pages/`（ChatPage、MemoryPage、GraphPage…）、`components/`、`layouts/`、`stores/`（Zustand 状态，目前主要是 authStore）。
- 对话页用 **SSE（Server-Sent Events）** 接收后端流式输出。

## 2. 核心链路：用户输入一句话后的执行生命周期

这是本文的重头戏。我们追踪**最核心的一条链路：用户在对话框敲下一句话 → AI 自主调用工具 → 流式返回带引用的答案**。

### 2.1 调用栈（从入口到内核）

```
前端 ChatPage  ──POST /api/chat/stream (SSE)──▶
  chat_controller.chat_stream()                       ← 路由层，返回 StreamingResponse
    └─ ChatService.stream_chat()                       ← 业务编排（核心！见下）
         ├─ 加载默认对话模型 + Agent 配置
         ├─ 按开关构建工具（知识库/记忆/联网）
         ├─ 判定强/弱模型 → 走两条路径之一：
         │     run_function_calling()   ← core/agent/orchestrator.py（强模型）
         │     run_react()              ← core/agent/orchestrator.py（弱模型）
         ├─ 工具执行时调用 core/rag.hybrid_search / core/memory.search_memory / web_search
         ├─ 逐 token 通过 SSE 吐给前端
         ├─ 落库 user / assistant 消息（带引用与工具调用元信息）
         └─ 回答后异步派发「对话记忆萃取」任务（不阻塞用户）
```

### 2.2 逐步拆解（对照 `services/chat_service.py` 的 `stream_chat`）

**Step 0｜入口**：`chat_controller.py:81` 的 `chat_stream` 把请求交给 `ChatService.stream_chat`，并以 `media_type="text/event-stream"` 返回 `StreamingResponse`——这就是流式输出的基础设施（`X-Accel-Buffering: no` 关掉 Nginx 缓冲，保证逐字吐出）。

**Step 1｜准备会话**：`_ensure_conversation` 找到或新建一条 `Conversation`，先把用户这句话作为 `Message(role=user)` 落库，并通过 SSE 的 `meta` 事件把 `conversation_id` 回传给前端。

**Step 2｜加载「人格与能力」**：
- `agent_repo.get_by_user` 取出该用户的 **Agent 配置**（temperature、system_prompt、三个工具开关 enable_knowledge / enable_memory / enable_web_search）；
- `build_default_chat_model` 按用户在「模型配置」里设的默认对话模型，实例化一个 LangChain `ChatOpenAI`（`core/llm/chat_model.py`）。

**Step 3｜构建工具集**（`_build_tools`）：这是 Agent「有哪些手」的地方。按「Agent 默认开关 + 本轮请求覆盖」决定启用哪些工具，每个工具是一个 LangChain `StructuredTool`：
- `build_knowledge_tool` → 内部调 `core/rag/search.py` 的 `hybrid_search`，并把命中的来源写进 `citations`（引用溯源）；
- `build_memory_tool` → 内部调 `core/memory/retrieval/searcher.py` 的 `search_memory`；
- `build_web_search_tool` → 内部调 `core/agent/web_search.py`。
> 关键技巧：工具用**闭包**捕获了 `session / user_id / citations`，所以工具被 LLM 触发时，能直接拿到「当前是谁、查到的引用往哪塞」——这是把「多租户上下文」注入工具的一种优雅写法。

**Step 4｜分流决策（方案 B 的核心）**：根据输入与模型能力，走四条分支之一（见 `chat_service.py:155-195`）：

| 条件 | 走哪条路 | 说明 |
|------|---------|------|
| 带图片 `image_keys` | `_stream_multimodal` | 多模态看图问答，**不走工具编排** |
| 没启用任何工具 | 纯 `model.astream` | 普通流式闲聊 |
| 模型支持 function_call | `run_function_calling` | **强模型**：原生工具调用 |
| 否则 | `run_react` | **弱模型**：提示词模拟 ReAct 降级 |

**Step 5｜工具循环（Agent 的心脏）** —— 看 `core/agent/orchestrator.py`：

- **强模型路径 `run_function_calling`**：`model.bind_tools(tools)` 后流式输出；若模型这一轮产生了 `tool_calls`，就**执行工具 → 把 `ToolMessage` 结果回灌进消息列表 → 继续下一轮**，直到模型不再要求调工具（视为最终答案）。最多 `MAX_TOOL_ITERATIONS = 5` 轮，防止死循环。
- **弱模型路径 `run_react`**：用 `react.jinja2` 提示词模板告诉模型按 `Thought / Action / Action Input` 格式输出，代码用正则把 `Action` / `Action Input` / `Final Answer` 解析出来，**手动调用工具，把 `Observation` 拼回对话**，再问下一轮。

两条路径**产出统一的事件流**，这是设计上的点睛之笔：
```python
{"type": "tool_call", "tool": ..., "query": ...}   # 我要调某工具
{"type": "token",     "text": ...}                  # 吐出一段文本
{"type": "final",     "text": ...}                  # 最终答案
```
上层 `ChatService` 不关心底层走哪条路，只消费这套统一事件 → 翻译成 SSE 事件吐给前端（`_event_to_sse`）。**「双实现、单接口」**，是值得学的解耦手法。

**Step 6｜收尾落库**：把累积的 `full_text` 作为 `Message(role=assistant)` 落库，`meta_data` 里带上 `citations`（引用）和 `tool_calls`（调了哪些工具，前端据此画图标），并 `touch` 会话更新时间，最后发 `done` 事件。

**Step 7｜自我成长（异步）**：`_dispatch_memory` 把用户这句话存成 `Memory(source=auto)`，然后 `extract_memory_task.delay(...)` **丢进 Celery 队列**——记忆萃取这种慢活在 worker 后台跑，绝不阻塞当前问答。这就是「Agent 用着用着越来越懂你」的实现入口。

### 2.3 另一条同样重要的链路：记忆是怎么「长」出来的（异步流水线）

问答只是「读」，记忆图谱的「写」发生在异步 worker 里（`tasks/memory.py` → `core/memory/extraction/orchestrator.py` 的 `run_extraction`），流水线为：

```
来源文本
  → 分块(chunker)
  → 陈述抽取(statement_extractor)
  → 三元组萃取(triplet_extractor)  ← 主谓宾 + 事件
  → 实体名向量化(embedder)
  → 批内去重 + 与图谱已有实体二层融合(dedup)
  → 组装四层溯源图 → 写 Neo4j
  → 增量社区聚类(label_propagation)
```

**四层溯源图**：`Dialogue(来源) → Chunk(片段) → Statement(陈述) → Entity(实体)`，实体之间再挂 `RELATION` 三元组边、`Event` 事件节点。这是阶段二「🧠 Memory」要细抠的部分，这里你只需建立印象：**记忆不是简单存原文，而是被 LLM 结构化成「可溯源、可去重、可聚类」的知识图谱**。

### 2.4 异步队列分工（`celery_app.py`）

| 队列 | 干什么 | 触发方式 |
|------|--------|---------|
| `parse` | 文档解析、图片描述 | 上传后派发 |
| `memory` | 记忆三元组萃取、去重 | 主动记住 / 对话后派发 |
| `beat` | 社区聚类、每日回顾 | Celery beat 定时（22:00 回顾、3:00 全量聚类兜底） |

> 设计取舍：**不自研调度器，直接用 Celery 标准多队列**——让「重活、慢活」与「在线问答」物理隔离，是 Agent 应用保证响应速度的关键工程手段。

## 3. 技术栈盘点

### 3.1 一张表看全

| 层 | 技术选型 | 在 Agent 里扮演的角色 |
|----|---------|----------------------|
| 前端 | React 18 + TypeScript + Ant Design 5 + Vite；Zustand；AntV X6 + ECharts | 对话 UI（SSE 流式）、知识图谱/时间线/仪表盘可视化 |
| 后端框架 | **FastAPI**（async）；依赖用 **uv** 管理 | 路由 + 流式响应（StreamingResponse + SSE） |
| 业务库 | PostgreSQL 16 + SQLAlchemy 2.0 (async) + Alembic | 用户/对话/消息/配置/记忆原文 |
| 向量/全文 | Elasticsearch 8.17（向量 + BM25 + **IK 中文分词**） | RAG 混合检索 |
| 记忆图谱 | Neo4j 5.26 | 实体-关系-事件三元组、四层溯源 |
| 异步/队列 | **Celery + Redis**（多队列 parse/memory/beat） | 解析、萃取、聚类、定时任务 |
| **LLM 编排** | **LangChain**（Agent 工具循环，「方案 B」） | 工具绑定、消息抽象、流式 astream |

### 3.2 关于「LangChain」用到了什么程度（学 Agent 最关心这个）

**它用 LangChain，但用得很「克制、很底层」**，没有用 LangChain 那些高度封装的 `AgentExecutor`/`create_react_agent` 黑盒，而是：
- 用 `langchain_openai.ChatOpenAI` 作为统一的模型客户端（因为五个 provider——OpenAI/通义/豆包/DeepSeek/智谱——都是 OpenAI 兼容协议，一个类就够，见 `core/llm/chat_model.py` 的注释）；
- 用 `langchain_core.tools.StructuredTool` 定义工具，用 `model.bind_tools()` 做原生 Function Calling；
- 用 `langchain_core.messages`（`SystemMessage / HumanMessage / AIMessage / ToolMessage`）手搭对话上下文；
- **工具循环（Tool Loop）是项目自己用一个 `for` 循环手写的**（`orchestrator.py`），不是 LangChain 替你跑的。

> 💡 这对你学习是**天大的好事**：项目把「Agent 工具循环」这层最核心的逻辑**自己实现并暴露在你眼前**（就 130 行的 `orchestrator.py`），你能清清楚楚看到「LLM 怎么决定调工具 → 工具结果怎么回灌 → 怎么收敛到最终答案」，而不是埋在框架黑盒里。这是理解 Agent 本质的最佳样本。
>
> 注意：项目里有**两套不同用途的 LLM 客户端**——问答链路用 LangChain 的 `ChatOpenAI`（`core/llm/chat_model.py`）；记忆萃取链路用项目自封装的 `LLMClient`（`core/llm/client.py`）。别混淆，阶段二会讲清各自场景。

### 3.3 几个值得记住的工程实践（生产级 Agent 的「卫生习惯」）

- **多租户隔离**：几乎所有业务表都带 `user_id`，工具、检索、图谱全部按 user 过滤。
- **密钥安全**：用户填的各家 LLM API Key 用 **Fernet 对称加密**存库，用时 `decrypt_secret` 解密。
- **配置驱动**：模型不是写死的，用户在「模型配置」里自己填 chat / embedding / multimodal / rerank / websearch，默认模型可切换。
- **强弱模型分流**：靠 `capability` 标记里有没有 `function_call` 决定走原生工具调用还是 ReAct 降级——**对弱模型友好**，是真实产品才会考虑的事。
- **启动自举**：`main.py` 的 lifespan 钩子在启动时自动建好 ES 索引和 Neo4j 约束/索引，无需手动初始化。

## 4. 给你的「阶段一」收尾与下一步

### 4.1 三句话总结

1. **架构**：前后端分离 + 四存储外置；后端是「controller→service→repository→model」严格单向分层，**`core/` 是 Agent 的大脑**（agent 编排 / rag 检索 / memory 图谱 / llm 客户端）。
2. **核心链路**：`/api/chat/stream` → `ChatService.stream_chat` → 构建工具 → **强模型 Function Calling / 弱模型 ReAct 二选一的工具循环** → 统一事件流 → SSE 流式吐字 + 落库 + 异步萃取记忆。
3. **技术栈**：FastAPI + LangChain（底层用法，工具循环自己手写）+ PostgreSQL/ES/Neo4j/Redis + Celery 多队列。

### 4.3 通往阶段二

> 准备好了就来问我阶段二，我们逐行啃 `orchestrator.py` 和 `core/memory/`。📚
