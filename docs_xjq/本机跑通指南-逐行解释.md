## 二、完整步骤（每行命令逐句解释）

### 第 0 步：处理三个拦路项

#### ① 启动 Docker 守护进程

```bash
docker info >/dev/null 2>&1 && echo OK || echo "还没起来"
```

逐段解释：

- `docker info`：查询 Docker 守护进程（daemon）的运行状态信息。守护进程没起时这条会报错。
- `>/dev/null`：把**标准输出**（正常信息）丢弃，我们只关心成功/失败，不需要看一大段输出。
- `2>&1`：把**标准错误**（fd 2）重定向到标准输出（fd 1）当前指向的地方（也就是 `/dev/null`），即错误信息也一并丢弃。
- `&& echo OK`：前一条命令**退出码为 0（成功）** 时，才执行 `echo OK`。
- `|| echo "还没起来"`：前一条命令**失败（非 0）** 时，执行这条提示。

> WSL2 下通常的做法：在 Windows 里打开 **Docker Desktop** 并确保它开启了 WSL 集成；启动后回到 WSL 终端再跑上面这条验证。

#### ② 安装 Node 18+（前端 npm 依赖）

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc
nvm install 20
node --version
```

逐行解释：

- `curl -o- <url> | bash`：用 `curl` 下载 nvm 的安装脚本。
  - `-o-`：把下载内容输出到**标准输出**（`-` 代表 stdout），而不是存成文件。
  - `| bash`：把下载到的脚本内容**通过管道直接交给 `bash` 执行**，完成 nvm 安装（nvm = Node Version Manager，Node 版本管理器）。
- `source ~/.bashrc`：重新加载当前 shell 的配置文件，使 nvm 安装脚本往 `.bashrc` 里追加的初始化代码**在当前终端立即生效**（否则要新开终端才有 `nvm` 命令）。
- `nvm install 20`：用 nvm 下载并安装 **Node.js 20**（满足项目「18+」要求的稳定 LTS 版本），安装后自动设为当前使用版本。
- `node --version`：打印 Node 版本号，确认安装成功（应显示 `v20.x.x`）。

#### ③ 查出谁占用了 8000 端口

```bash
lsof -i :8000
```

- `lsof`：list open files，列出「打开的文件」。在 Linux 里网络连接也是文件。
- `-i :8000`：筛选条件，只看**占用了 8000 端口**的进程。
- 输出里的 `PID` 列就是占用进程号。确认无用后可 `kill <PID>` 关掉；若是重要进程，则改用「后端换端口」方案（见第 3 步末尾）。

> 若没有 `lsof`，可用等价命令：`ss -ltnp | grep :8000`（`-l` 监听中、`-t` TCP、`-n` 不解析端口名、`-p` 显示进程）。

### 第 1 步：创建根目录 `.env`（供 docker-compose 读取）

```bash
cp .env.example .env
```

- `cp`：copy，复制文件。
- `.env.example`：仓库里提供的**环境变量模板**（已 git 跟踪，可安全提交）。
- `.env`：复制出的实际配置文件（已被 `.gitignore` 忽略，不会提交，可放密码）。
- 作用：docker-compose 启动存储容器时会读这个 `.env` 里的 `POSTGRES_PASSWORD`、`NEO4J_PASSWORD` 等。**默认值即可跑通，无需修改**。

---

### 第 2 步：启动四个存储容器

```bash
docker compose up -d postgres elasticsearch neo4j redis
docker compose ps
```

第一行逐段解释：

- `docker compose up`：根据 `docker-compose.yml` 创建并启动服务容器。
- `-d`：detached（后台）模式运行，容器在后台跑，终端不被占用。
- `postgres elasticsearch neo4j redis`：**只启动这四个存储服务**，不启动应用容器（应用我们要在本机裸跑，方便热重载和打断点调试）。

第二行：

- `docker compose ps`：列出 compose 管理的容器及其状态（`running` / `healthy` 等）。
- ES（Elasticsearch）首次启动会自动 build 内置 IK 中文分词插件的镜像，需几分钟；启动也较慢（约 30~60 秒）。**等四个都健康再进行下一步**。

---

### 第 3 步：配置并启动后端

```bash
cd api
uv sync
cp .env.example .env
```

逐行解释：

- `cd api`：进入后端目录（FastAPI 项目根，含 `pyproject.toml`）。
- `uv sync`：uv 读取 `pyproject.toml` / `uv.lock`，**自动完成三件事**：
  1. 若本机没有满足 `requires-python>=3.12` 的解释器，**自动下载 Python 3.12**；
  2. 在 `api/.venv` 创建虚拟环境；
  3. 按锁文件安装所有依赖（精确版本，可复现）。
- `cp .env.example .env`：复制后端环境变量模板为实际配置文件（同样被 gitignore 忽略）。

编辑 `api/.env`，**只有这两项必须改**（其余默认值指向 localhost，与第 2 步容器端口一致）：

```dotenv
JWT_SECRET=随便敲一长串随机字符
FERNET_KEY=<下面命令生成的值>
```

- `JWT_SECRET`：JWT（登录令牌）的签名密钥。任意一段足够长的随机字符串即可，泄露会导致别人能伪造登录态。
- `FERNET_KEY`：用于**对称加密用户填的 LLM API Key**（Fernet 算法）。**格式必须是合法的 44 位 base64**，不能手敲乱填，否则后端启动/加密时直接报错。

生成 `FERNET_KEY`：

```bash
uv run python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

- `uv run python`：在 uv 管理的虚拟环境里运行 Python（无需手动 `activate`）。
- `-c "<代码>"`：直接执行这段 Python 代码字符串。
- `from cryptography.fernet import Fernet`：导入加密库的 Fernet 类。
- `Fernet.generate_key()`：生成一把符合格式要求的随机密钥（bytes）。
- `.decode()`：把 bytes 解成字符串，方便复制粘贴进 `.env`。
- `print(...)`：打印出来。把这串复制到 `FERNET_KEY=` 后面。

建表并启动后端：

```bash
uv run alembic upgrade head
uv run python run.py
```

- `uv run alembic upgrade head`：运行 Alembic 数据库迁移。
  - `alembic`：SQLAlchemy 配套的数据库版本迁移工具。
  - `upgrade head`：把数据库结构**升级到最新版本**（`head` = 迁移链最末端），即按 `migrations/` 里的脚本把所有表建好。
- `uv run python run.py`：启动后端服务（`run.py` 是本地启动入口，内部拉起 FastAPI / uvicorn）。

> **若 8000 端口腾不出来**：可在 `api/.env` 中改端口配置（变量名以 `api/config.py` 实际读取的为准，常见为 `APP_PORT` / `PORT`），同时把前端 `web/vite.config.ts` 里 `/api` 代理目标端口一并改掉，保持一致。

验证后端就绪：

```bash
curl http://localhost:8000/api/health
curl http://localhost:8000/api/hello
```

- `curl <url>`：命令行 HTTP 客户端，向后端发 GET 请求。
- `/api/health`：健康检查接口，正常时四个存储应全部返回 `"ok"`。
- `/api/hello`：欢迎接口，能返回信息说明后端进程活着。

---

### 第 4 步：启动 Celery worker（**新开一个终端**，在 `api` 目录）

```bash
uv run celery -A app.celery_app.celery_app worker -l info -Q default,parse,memory,beat
```

逐段解释：

- `uv run celery`：在虚拟环境里运行 celery 命令。
- `-A app.celery_app.celery_app`：`-A` = application，指定 Celery 应用实例的导入路径（`app/celery_app.py` 里的 `celery_app` 对象）。
- `worker`：以「工作进程」模式启动，负责**消费队列里的异步任务**。
- `-l info`：`--loglevel info`，日志级别设为 info，能看到任务收发与「记忆萃取完成」等关键日志。
- `-Q default,parse,memory,beat`：`--queues`，指定本 worker 监听的队列。
  - `parse`：文档解析/分块/向量化任务；
  - `memory`：记忆萃取（抽三元组、建图谱）任务；
  - `beat`：定时任务被投递到的队列；
  - `default`：兜底队列。
- **作用**：不启动 worker 的话，上传文档、记忆萃取会永远卡在「处理中」。

> 你是 Linux/WSL 环境，**无需** Windows 才要的 `--pool=solo`；想提高并发可加 `--concurrency=4`。
> Celery **beat**（每日回顾、定时聚类）是可选的，MVP 阶段可不起，不影响问答与记忆萃取。

### 第 5 步：启动前端（**再开一个终端**）

```bash
cd web
npm install
npm run dev
```

- `cd web`：进入前端目录（React + TypeScript + Vite）。
- `npm install`：读取 `package.json` 安装所有前端依赖到 `node_modules`（首次较慢）。
- `npm run dev`：运行 `package.json` 中的 `dev` 脚本，启动 Vite 开发服务器（带热更新）。
- 启动后访问 `http://localhost:5173`。前端已配 `/api` 代理转发到后端 8000，无需额外处理跨域。

---

### 第 6 步：注册账号并配置模型（在前端 UI 里完成，不写在 .env）

1. 打开前端 → 点「注册」创建账号 → 登录。
2. 进入 **设置 → 模型配置**，至少添加两个模型：
   - **对话模型**（`type=chat`）：如 DeepSeek `deepseek-chat`，填 base_url 与 API Key。强模型建议勾上 `function_call` 能力 → 问答走原生 Function Calling（强模型路径）。
   - **Embedding 模型**（`type=embedding`）：如智谱 `embedding-3`。**维度必须为 1024**，要与后端 `EMBEDDING_DIMS=1024` 及 ES 索引维度一致，否则向量写入/检索报「维度不匹配」。
3. 每个模型添加后点「测试连接」，通过后「设为默认」（后端靠 `is_default` 选默认模型）。

> 为什么模型配置不放 `.env`？因为这是**多用户**系统，每个用户用自己的 key 和模型，配置加密存进 PostgreSQL 的 `model_configs` 表（API Key 用上面的 `FERNET_KEY` 加密）。

## 五、常见卡点速查

| 现象 | 原因 | 处理 |
|------|------|------|
| `/api/health` 某存储连不上 | 容器没起/没健康/端口被占 | `docker compose ps` 看状态，ES 多等一会儿 |
| 文档/记忆一直「处理中」 | worker 没起或队列不对 | 确认第 4 步 worker 在跑，队列含 `parse,memory` |
| 对话报「未配置对话模型」 | 没配/没设默认 | 模型配置里添加并「设为默认」 |
| 向量写入报维度不匹配 | Embedding 维度 ≠ 1024 | 换成 1024 维模型（如智谱 `embedding-3`） |
| 中文检索效果差 | 误用官方 ES 镜像（无 IK 分词） | 删 ES 容器与卷，`docker compose up -d --build elasticsearch` |
| 改了 ORM 模型 | 表结构变了 | `uv run alembic revision --autogenerate -m "说明"` 后 `upgrade head` |

## 六、实际跑通实录（本机一次真实跑通的全过程）

> 本节是 2026-06-05 在这台 WSL2 机器上**真实跑通一遍**的完整记录：每一步实际怎么做、踩到哪些坑、怎么解决。
> 与前面的「标准步骤」相比，这里强调的是**现实里不会一帆风顺**——本次就遇到了 3 个拦路问题 + 2 个模型适配问题，下面如实复盘。

### 6.0 总体结果

最终 7 个服务全部就绪、`/api/health` 四存储全 `true`，并用「记忆萃取 → 对话问答」端到端冒烟测试通过（Agent 自动调 `memory_search` 答出「多多」）。打开 `http://localhost:5173`，用脚本创建的开发账号 `comet` / `comet123` 即可登录使用。

### 6.1 开工前体检：发现了什么

先探测环境，结论是「大体齐全，但有坑」：

| 依赖 | 实测结果 | 处理 |
|------|---------|------|
| Docker / Compose | 已装（29.3.1 / v5.1.1），但**守护进程要 sudo 才能访问** | socket 是 `root:root`，当前用户不在可访问组 → 起容器须 sudo |
| uv | 0.10.5，OK | — |
| Python | 系统 3.11，conda 环境 `background_estimate` 也是 3.11 | 不用它们，让 uv 自己管版本（但后面踩坑，见 6.3） |
| Node / npm | **未安装** | 用 nvm 装 Node 20 |
| 端口 8000 | **被另一个项目占用**（`LangGraph-trip-planner/backend`） | 杀掉那个进程腾出端口 |
| 5432/9200/7474/7687/6379/5173 | 空闲 | — |

> 关键判断手法：用 `ls -l /proc/<PID>/cwd` 查出占用 8000 的进程工作目录，确认它是**另一个项目**（不是 Comet 的残留），才敢杀。

### 6.2 拦路问题①：8000 端口被别的项目占用

先查父进程，确认杀掉不会被自动拉起（父进程只是个终端 bash，不是 reloader）：

```bash
ps -p <PID> -o pid,ppid,cmd      # 看进程身份
ls -l /proc/<PID>/cwd            # 看它属于哪个项目
kill <PID>                       # 普通 kill 即可，已成功释放
```

**做法说明**：先「验明正身」再动手——确认是无关项目、确认父进程不会重启它，才执行 `kill`。释放后 Comet 后端就能用默认 8000，无需改端口配置。

### 6.3 拦路问题②：uv 默认选了 Python 3.14，依赖编译失败

`uv sync` 第一次跑，uv 自动挑了**满足 `>=3.12` 的最新版 3.14.3**，结果 `pydantic-core` 没有 3.14 的预编译 wheel，回退到源码用 maturin/Rust 编译，**失败**：

```
Error: command ['maturin', 'pep517', 'build-wheel', ...] returned non-zero exit status 1
help: `pydantic-core` (v2.27.2) was included because ... depends on `pydantic` (v2.10.4)
```

> ⚠️ 一个隐蔽的坑：当时命令是 `uv sync 2>&1 | tail -20`，**管道末端 `tail` 的退出码是 0，把 uv 的失败掩盖了**。是靠「`import cryptography` 报 ModuleNotFoundError」才发现根本没装上。教训：判断 `uv sync` 成功不能只看 `$?`，要实际 import 验证。

**解决**：把 Python 钉到 3.12（所有钉死版本的依赖都有 3.12 wheel），删掉 3.14 的 venv 重建：

```bash
cd api
uv python pin 3.12          # 写入 .python-version，并自动下载 cpython-3.12.12
rm -rf .venv                # 删掉用 3.14 建坏的虚拟环境
uv sync                     # 用 3.12 重新装，成功
```

验证这次是真的装好了（不要只信退出码）：

```bash
.venv/bin/python -c "import cryptography, fastapi, alembic, celery, sqlalchemy, uvicorn, pydantic_core; print('OK')"
```

> 顺带说明 conda 环境：用户用的 `background_estimate` 是 Python 3.11，**项目要求 3.12+ 且强制用 uv**，所以没有用 conda 环境——uv 自带的 3.12 venv 是自洽的，与 conda 无关。

### 6.4 拦路问题③：Docker 要 sudo，且 sudo 需要密码

`docker info` 报 `permission denied ... /var/run/docker.sock`；socket 属主是 `root:root`，当前用户也不在 docker 组；`sudo` 又需要交互式密码。

**做法**：这一步天然需要用户参与（密码只有用户能给）。拿到密码后，用 `sudo -S`（从标准输入读密码）非交互执行，并放到后台跑（ES 首次要拉镜像 + 构建，耗时几分钟）：

```bash
cd /home/jingqinxue/workspace/Comet
echo '<密码>' | sudo -S docker compose up -d postgres elasticsearch neo4j redis
```

启动后**轮询健康状态**，等三个带健康检查的容器都 `healthy` 再继续（用 `docker inspect` 取 `.State.Health.Status`）：

```bash
for i in $(seq 1 30); do
  pg=$(echo '<密码>' | sudo -S docker inspect --format '{{.State.Health.Status}}' comet-postgres)
  es=$(echo '<密码>' | sudo -S docker inspect --format '{{.State.Health.Status}}' comet-es)
  neo=$(echo '<密码>' | sudo -S docker inspect --format '{{.State.Health.Status}}' comet-neo4j)
  [ "$pg" = healthy ] && [ "$es" = healthy ] && [ "$neo" = healthy ] && break
  sleep 5
done
```

> 本次四个容器很快就 healthy 了（镜像已在本地缓存）。

### 6.5 配 env：两项密钥怎么填进去的

两个 `.env` 都从模板复制，再用脚本精确替换两项必改密钥（避免 sed 对特殊字符的转义问题）：

```bash
# 根 .env（docker-compose 用，默认值即可）
cp .env.example .env

# api/.env
cd api && cp .env.example .env
# JWT_SECRET：用 secrets 生成 96 位 hex
python3 -c "import secrets; print(secrets.token_hex(48))"
# FERNET_KEY：用项目自己的 cryptography 生成合法 44 位 base64
.venv/bin/python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

> 替换用的是 Python 读文件 `.replace("JWT_SECRET=change-me-in-production", ...)` 再写回，比 `sed` 稳。`FERNET_KEY` 必须等 `uv sync` 装好 cryptography 后才能生成——所以顺序上它排在依赖安装之后。

### 6.7 模型适配问题：HepAI 的两个坑

模型配置本该在前端 UI 点，但本次为了端到端验证，**直接调后端 API 脚本化完成**（注册 → 登录拿 token → POST `/api/models` 建配置 → POST `/api/models/{id}/test` 测连接）。配置字段取值是先读代码确认的：

- `type`: `chat` / `embedding`；`provider`: 填 `openai`（HepAI 是 OpenAI 兼容接口）；
- `base_url`: `https://aiapi.ihep.ac.cn/apiv2`；`capability`: chat 勾 `["function_call"]`。

**坑 A —— HepAI 的 Claude 模型当前返回 500。** 用 key 直接打 HepAI 验证：

```
[anthropic/claude-opus-4-7] HTTP 500   ← 不可用（服务端报错，非配置问题）
[claude-opus-4-7]           HTTP 500   ← 同样不可用
[openai/gpt-4o-mini]        OK         ← 可用，支持 function calling
[deepseek-ai/deepseek-v4-pro] OK
```

→ **对话模型改用 `openai/gpt-4o-mini`**（同一个 key），等 HepAI 的 Claude 恢复再在 UI 改回。

**坑 B —— embedding 维度必须 1024。** 查 HepAI 的 `/models`，OpenAI 系 embedding 是 1536/3072，都不匹配；只有 **`hepai/bge-m3:latest` 是 1024 维**，正好等于 `EMBEDDING_DIMS=1024`。→ embedding 模型选 bge-m3，测试连接通过。

> 排查手法：不要在应用层瞎猜，**拿 key 直接 `curl` HepAI 的 `/chat/completions` 和 `/models`**，一眼看出哪个 model 名可用、哪个 embedding 是 1024 维。

### 6.8 冒烟测试：端到端验证整条链路

```
① POST /api/memories/remember   text="我在腾讯做后端，养了只叫多多的小狗…"
   → worker 异步萃取，日志：记忆萃取完成 statements=3 entities=5 relations=4 events=1，社区聚类完成

② POST /api/chat/stream         message="我养了什么宠物？它叫什么名字？"  enable_memory=true
   → SSE 事件流：meta → tool_call{memory_search, query:"宠物"} → token×13 → done
   → 拼出回答：「你养了一只小狗，它的名字叫多多。」 ✅
```

跑通这条 = **Agent 工具循环 + 记忆萃取 + 向量检索 + HepAI 模型** 整条链路都通了。

> 小插曲：验证脚本里把 `多多` 误判为「未命中」，其实是 SSE 的 token 内容是 `{"text":"多"}` 这种再包了一层 JSON，脚本没解开 `text` 字段而已——真实回答是对的。说明**校验脚本本身也要对着真实数据格式写**。

### 6.9 本次踩坑总结（一句话版）

1. **8000 被别的项目占** → 先 `ls /proc/PID/cwd` 验明正身再 `kill`。
2. **uv 选 3.14 装不上** → `uv python pin 3.12` 后 `rm -rf .venv && uv sync`；且别信管道退出码，要 import 验证。
3. **Docker 要 sudo** → `echo 密码 | sudo -S docker compose ...`，再轮询 health。
4. **HepAI Claude 500** → 改 `openai/gpt-4o-mini`。
5. **embedding 必须 1024 维** → 选 `hepai/bge-m3:latest`。
