# Comet（彗记）项目介绍与技术栈

> 个人 AI 知识库 + 记忆助手。本文是项目的总览说明，适合用于项目自述、简历描述与面试开场介绍。

---

## 一、一句话介绍

Comet 是一个多用户的个人 AI 知识库与记忆助手：把用户的文档、图片、网页沉淀为可语义检索的知识库，从对话中自动萃取「记忆」构建专属知识图谱，并用 LLM Agent 自主编排「知识库 / 记忆 / 联网 + 可外接的 MCP 工具」来回答问题。更进一步，它会分析用户的对话情绪、把记忆深化为有动力学属性的分层记忆，并据此做情绪化音乐推荐。前后端分离。

面试自我介绍可以这样说：

> 我做了一个个人 AI 知识库 + 记忆助手。它有三个核心能力：一是 RAG 知识库，支持文档/图片/网页入库、中文混合检索带引用；二是记忆系统，从对话里自动萃取三元组写进 Neo4j 知识图谱，区分画像和事件；三是智能问答，把检索能力做成 LangChain Agent 工具，让大模型自主决定调哪个工具，支持 SSE 流式输出。整体是 FastAPI + React，用到了 PostgreSQL、Elasticsearch、Neo4j、Redis 四种存储。

## 二、技术栈

| 层 | 选型 | 说明 |
|----|------|------|
| 前端 | React 18 + TypeScript + Ant Design 5 + Vite | 状态 Zustand；图谱 AntV X6；图表 ECharts |
| 后端 | FastAPI（异步） | 分层架构 controller→service→repository→model/db |
| 业务库 | PostgreSQL 16 + SQLAlchemy 2.0(async) + Alembic | 用户、配置、文档元数据、会话、记忆原文等 |
| 向量/全文 | Elasticsearch 8.17（IK 中文分词） | 向量召回 + BM25，混合检索 |
| 记忆图谱 | Neo4j 5.26 | 实体-关系-事件三元组，向量索引 + 全文索引 |
| 异步/缓存 | Celery + Redis | 多队列：解析 / 萃取 / 定时任务 |
| LLM 编排 | LangChain + langchain-openai | Agent 工具循环（方案B）；MCP 工具接入用 langchain-mcp-adapters |
| 依赖管理 | 后端 uv，前端 npm | — |

**为什么用四种存储**（面试高频）：

- **PostgreSQL**：结构化业务数据，强一致、事务、关系约束。
- **Elasticsearch**：知识库需要「向量语义检索 + BM25 关键词检索」，ES 8 原生支持 dense_vector + kNN，又是成熟的全文引擎，一个存储满足两种召回。
- **Neo4j**：记忆是「实体-关系」结构，本质是图。用图数据库做多跳关系遍历（一跳邻居拼上下文）天然契合，比在关系库里自连接高效直观。
- **Redis**：既是缓存，又作 Celery 的 broker / result backend。

## 三、功能模块

```
┌─────────────────────────────────────────────────────────┐
│                      前端 (React)                         │
│  登录 / 知识库 / 图片库 / 记忆 / 对话 / 搜索 / 图谱 / 仪表盘 │
└───────────────────────────┬─────────────────────────────┘
                            │ REST + SSE
┌───────────────────────────▼─────────────────────────────┐
│                     后端 (FastAPI)                        │
│  ┌──────────┬──────────┬──────────┬──────────┬─────────┐ │
│  │ 账号鉴权  │ 模型配置  │ 知识库RAG │ 记忆系统  │ 智能问答 │ │
│  ├──────────┴──────────┴──────────┴──────────┴─────────┤ │
│  │   搜索导航 / 收藏 / 标签 / 社区聚类 / 可视化统计       │ │
│  └───────────────────────────────────────────────────── │
└───┬──────────┬──────────┬──────────┬─────────────────────┘
    ▼          ▼          ▼          ▼
PostgreSQL   ES       Neo4j      Redis ─── Celery worker/beat
```

1. **账号体系**：注册/登录/JWT(access+refresh)/bcrypt/数据隔离。所有业务表带 `user_id` 做多租户隔离。
2. **模型配置**：用户自配多 provider 的对话/向量/多模态/重排/联网模型；API Key 用 Fernet 加密入库、接口返回掩码；支持连接测试与设默认。
3. **知识库 RAG**：文档（PDF/Word/MD/TXT/HTML）/网页/图片入库，父子分块 + IK 中文分词，向量+BM25 混合检索（可选 rerank），带引用溯源；图片多模态识别；AI 自动打标签。
4. **记忆系统**：从「主动记住」或对话异步萃取三元组写入 Neo4j 四层溯源图谱（来源→片段→陈述→实体），区分画像实体与事件（带时间），两层去重，混合检索。
5. **智能问答**：知识库/记忆/联网三个工具做成 LangChain Agent，强模型走原生 function calling、弱模型走 ReAct 降级；SSE 流式；带引用与工具调用标记；多模态看图问答；回答后异步萃取记忆。
6. **搜索与导航**：全局搜索（文档+图片+记忆三排并列，语义相关度门控）、收藏夹、标签管理、每日回顾。
7. **社区聚类**：标签传播算法（LPA）把记忆实体聚成主题社区，增量+定时双触发。
8. **可视化统计**：知识图谱（AntV X6）、事件时间线、统计仪表盘（ECharts）。
9. **能力增强（v0.0.2）**：消息反馈/重新生成；工具体系开放化（内置工具注册中心 + MCP 接入，官方 langchain-mcp-adapters，SSE / Streamable HTTP）；提示词优化器。
10. **情绪记忆（v0.0.2）**：每轮对话后异步分析用户情绪，valence-arousal 二维 + 滚动画像；为音乐推荐供数。
11. **记忆深化与分层（v0.0.2）**：记忆节点加 importance/confidence/access_count/mention_count 等动力学属性，短期/长期分层 + 定时巩固（只升不降，记忆不遗忘）+ 高频实体 LLM 画像增强。
12. **情绪化音乐推荐（v0.0.2）**：按「记忆偏好歌手 + 当前情绪」打分选歌；三层音源降级（本地上传 > 咪咕免费现播现取 > 仅展示）；歌曲处理异步化；旋转 CD + 滚动歌词的沉浸式悬浮播放器。

## 四、架构设计要点（面试可讲）

### 1. 严格分层

`controller → service → repository → model/db` 单向调用：

- controller 只做路由、入参校验、调 service、包装响应，不写业务逻辑；
- service 写业务逻辑、编排 repository 与外部调用；
- repository 只做数据存取，不写业务规则。

横切能力（RAG / 记忆 / Agent / LLM / 存储）放在 `core/` 下按子系统分目录，子系统内部再按流水线阶段拆（如记忆分 preprocessing / extraction / retrieval / clustering）。

### 2. 异步任务解耦

耗时操作（文档解析、记忆萃取、社区聚类、情绪分析、歌曲处理）走 Celery 异步队列，接口立即返回。队列按域拆分：`parse`（解析 / 图片 / 歌曲处理）/ `memory`（萃取 / 情绪）/ `beat`（定时：每日回顾、社区聚类、记忆巩固）。

> 踩坑点：Celery 任务是同步入口，内部用 `asyncio.run` 跑异步流程；每个任务用任务级独立 DB 引擎（NullPool）+ 重置 ES/Neo4j 客户端，避免全局单例绑定到已关闭的事件循环。Windows 上 worker 必须用 `--pool=solo`（prefork 有权限问题）。

### 3. 统一响应与异常

所有接口返回 `{ code, message, data }`；业务失败抛 `BizError`（带中文提示 + 业务码 + HTTP 码），全局异常处理器兜底。

### 4. 安全

- 密码 bcrypt；JWT 分 access(30min)/refresh(7d)，payload 带 type 区分。
- API Key 等敏感信息 Fernet 对称加密存储，接口返回掩码。
- 多租户隔离：所有查询强制带 user_id。
- 网页导入做 SSRF 防护。

## 六、常见追问（进阶）

**Q：为什么不用现成的 RAG/记忆框架（Dify、LangChain memory 等），要自己写？**
现成框架封装度高、定制难，且很多能力是黑盒。自研能完全掌控分块策略、混合检索权重、记忆萃取的受控词表和去重逻辑、Agent 的双路径，也更能体现对原理的理解（面试加分）。借鉴了开源项目的思路但按需求重写、砍掉企业级冗余。

**Q：项目的数据流是怎样的？**
写入侧：上传/对话 → 异步任务（解析/萃取）→ 写 ES（知识库）/Neo4j（记忆）。读取侧：提问 → Agent 编排工具 → 检索 ES/Neo4j → 拼上下文 → LLM 流式生成 → 落库会话 → 回答后异步萃取记忆（闭环）。

**Q：四个存储之间怎么协同的？数据一致性？**
各司其职无强一致需求：PG 存元数据和原文（事务保证），ES 存向量/全文，Neo4j 存图谱，Redis 做队列/缓存。跨存储不用分布式事务——异步任务里出错就把 PG 状态置 failed，ES/Neo4j 的写入用幂等（删旧重写 / MERGE）保证可重试，最终一致即可。

**Q：如果让你重构，会改什么？**
①检索融合可上 RRF 提鲁棒；②会话历史加摘要压缩；③记忆冲突可加时序消解；④图谱可视化大数据量做聚合加载；⑤加可观测性（链路追踪、检索质量埋点）。

**Q：项目体现了你哪些能力？**
①体系化：四存储选型、分层架构、异步解耦；②AI 工程：RAG 全链路、Agent 编排、知识图谱萃取，懂原理也能落地；③工程素养：统一响应/异常、连接池、安全（加密/隔离/SSRF）、踩坑解决（事件循环、实体去重 bug）。
