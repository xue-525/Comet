# 记忆模块 — 设计与面试八股

> 结合 Comet 记忆系统的真实实现，讲解知识图谱 / 三元组萃取 / 实体去重 / 图检索 / 社区聚类 等面试高频点。
> 对应代码：`api/app/core/memory/`（ontology/graph_models/graph_schema/preprocessing/extraction/retrieval/clustering）+ `api/app/repositories/neo4j/`

---

## 一、模块在做什么

让助手「记住」关于用户的长期信息。从「主动记住」的文本或对话中，自动萃取出**实体-关系-事件**三元组，写入 Neo4j 知识图谱，构建用户的专属画像与经历时间线，并支持语义检索。这就是给 AI 装上「长期记忆」。

和 RAG 的区别：RAG 存的是**文档片段**（非结构化文本块），记忆存的是**结构化的实体关系图**。RAG 回答「文档里写了什么」，记忆回答「关于用户本人的事实」。

---

## 二、四层溯源图谱结构

```
Dialogue（来源：一次对话/一段主动记住的文本）
  └─[:HAS_CHUNK]→ Chunk（片段）
        └─[:HAS_STATEMENT]→ Statement（原子陈述句，带类型/时间）
              └─[:MENTIONS]→ Entity（实体）

在此之上的语义层：
  Entity ─[:RELATION{predicate}]→ Entity   实体间三元组关系
  Event  ─[:INVOLVES]→ Entity              事件涉及的实体（带 event_time，供时间线）
  Entity ─[:IN_COMMUNITY]→ Community        社区聚类
```

**为什么保留四层溯源**：每个实体都能回溯到「它是从哪段对话、哪个片段、哪句话萃取出来的」，保证可解释、可审计、可纠错。这是企业级记忆系统的做法。

**两类记忆**：
- **画像类（Entity）**：稳定的偏好/关系/属性（「在腾讯工作」「喜欢周杰伦」）。
- **事件类（Event）**：一次性、有明确时间的经历（「2025年3月看了演唱会」），带 event_time，用于时间线。

## 三、萃取流水线（写入侧，核心）

`core/memory/extraction/orchestrator.py`，异步执行：

```
一段文本
  → 1. 分块（~512 token）
  → 2. 原子陈述抽取（statement_extractor）
        每句标注：陈述类型(FACT/OPINION/PREDICTION/SUGGESTION)
                 时间类型(STATIC/DYNAMIC/ATEMPORAL)
                 指代是否未解析（未解析则跳过）
  → 3. 三元组萃取（triplet_extractor）
        从陈述抽 实体 + (主语,谓词,宾语) + 事件
        受控词表约束：13 类实体 + 13 类谓词
  → 4. 实体 name 向量化
  → 5. 两层去重
  → 6. 组装四层溯源图模型
  → 7. 单事务原子写 Neo4j
  → 8. 增量社区聚类
```

**受控词表（ontology）**：13 类实体类型（生命体/组织/地点设施/知识能力/偏好习惯…）+ 13 类谓词（拥有/位于/偏好/了解/负责…），用中文标签约束 LLM 输出，避免实体类型泛滥、关系碎片化。LLM 越界输出会被规范化到「其他」/「关联于」。

**提示词工程**：用 jinja2 模板（`prompts/*.jinja2`，纯中文），把受控词表、示例、输出 JSON schema 渲染进 prompt。关键技巧：指代消解（「我」统一为「用户」）、不可解析的指代整条跳过、要求严格 JSON 输出。

## 四、两层实体去重（面试亮点）

**问题**：同一实体多次出现/多次萃取会产生重复节点（如「用户」反复出现）。

**两层去重**（`extraction/dedup.py`）：

### 第一层：批内去重
本次萃取出的实体之间：
- 同名同类型 → 直接合并（不问 LLM）；
- 名称相似（文本相似度 ≥ 0.8）/ 向量相似 / 包含关系 → 交给 LLM 判定是否同一实体，置信度 ≥ 0.8 才合并。
- 合并产生 id 重定向表（old_id → canonical_id）。

### 第二层：与图谱已有实体融合
去重后的实体再和 Neo4j 里同类型的已有实体比对：
- **同名同类型 → 直接复用已有节点 id（快路径，不问 LLM）**；
- 相似的交给 LLM 判定，命中则复用已有 id 并合并别名/描述。

> **踩坑修复**：早期第二层即使同名也每次问 LLM，而 LLM 非确定性 + prompt 倾向「拿不准不合并」，导致「用户」跨多次萃取被反复判为不同 → 重复建节点。修复是加同名快路径（同名同类型直接复用，不问 LLM），并提供历史重复合并接口（按同名同类型分组，把重复节点的边改接到保留节点后删除）。

去重为什么用「规则初筛 + LLM 精判」：全量两两问 LLM 太慢太贵；先用文本/向量相似度初筛候选对，只对可疑对问 LLM，兼顾效率和准确。

## 五、图谱混合检索（检索侧）

`core/memory/retrieval/searcher.py - search_memory`：

```
1. 实体向量召回：query 向量化 → Neo4j 向量索引 kNN
2. 全文召回：Neo4j cjk 全文索引（中文分词）
3. 归一化 + 加权融合：0.6 向量 + 0.4 全文
4. 一跳邻居遍历：命中实体取其 RELATION 关系，拼成「实体 + 关联事实」上下文
```

和 RAG 检索同款混合策略（向量+全文+融合），区别是多了**一跳邻居关系遍历**——这是图数据库的优势：召回实体后顺着关系边把关联事实一起带出来（「用户 — 养 → 多多」），给 LLM 的上下文更结构化。`format_memory_context` 把结果拼成文本供问答 Agent 的记忆工具复用。

Neo4j schema（`graph_schema.py`）：节点 id 唯一约束（保证 MERGE 幂等）、user_id 属性索引、cjk 全文索引、1024 维 cosine 向量索引、记忆层级（memory_layer）属性索引，启动时幂等创建。

## 五点五、记忆动力学与分层巩固（v0.0.2 新增亮点）

让记忆从「扁平的事实集合」进化为有**轻重、亲疏、冷暖**的活体。核心是给节点加动力学属性 + 短期/长期分层 + 定时巩固。

### 1. 节点动力学属性（`graph_models.py`）

- **通用**（Entity/Statement/Relation）：`importance` 重要度（LLM 萃取时评分）、`confidence` 置信度、`access_count` 被检索命中次数（检索侧回写）、`last_access_at` 最近命中时间、`memory_layer` 记忆层级（short_term / long_term）。
- **实体专属**：`mention_count` 被提及次数（每次萃取命中 +1）、`connect_strength` 连接强度、`core_facts` 核心事实、`traits` 特质标签（后两者由巩固时 LLM 画像增强产出）。

属性来源「双轨」：①LLM 萃取时直接给 importance/confidence 评分；②检索命中、重复提及在运行中累积 access_count / mention_count。

### 2. 检索融合排序引入重要度（`searcher.py`）

在「向量召回 + 全文召回」的归一化融合基础上，加入重要度加权与长期记忆加成：

```
score = 0.55·向量norm + 0.30·全文norm + 0.15·importance
长期记忆额外 +0.05
```

命中后回写 access_count + last_access_at（失败不影响检索），让「更重要、更稳定、更常被用到」的记忆优先浮现——这本身又反哺了巩固判定。

### 3. 短期 → 长期 巩固（`consolidation/consolidator.py` + beat 每日 4:00）

新萃取实体默认 `short_term`。定时巩固任务把满足**任一**条件的短期记忆提升为 `long_term`（**只升不降**）：

- `access_count >= 阈值`（经常被检索复用）；
- `importance >= 阈值`（本身重要）；
- `mention_count >= 阈值` 且已存在满 N 小时（反复被提及且沉淀了一段时间）。

提升后再对 **top-K 高频长期实体**做「画像增强」：用 LLM 汇总其关联陈述，回写 `core_facts`（核心事实）与 `traits`（特质标签）。也提供手动入口 `POST /memories/consolidate`。

### 4. 「记忆不遗忘」原则

全程**只升不降、不删不衰减**——这是与本项目刻意砍掉的 ACT-R / 艾宾浩斯遗忘引擎的根本区别。记忆层级单向提升，重要度/访问次数只增不减，保证用户信息长期留存。前端记忆画像页展示「长期」标签、特质标签与核心事实，并提供「记忆巩固」按钮。

> 面试一句话：我给记忆图谱节点加了 importance/confidence/access_count/mention_count 等动力学属性和 short/long 双层；检索按重要度加权排序、命中回写访问统计；定时巩固按「访问/重要度/提及」任一达标把短期记忆只升不降地提升为长期，并对高频实体做 LLM 画像增强补充核心事实与特质。遵循「记忆不遗忘」，只升不降。

## 六、社区聚类（标签传播 LPA）

`core/memory/clustering/label_propagation.py`：把语义相关的实体自动聚成「主题社区」。

```
全量聚类：
  每个实体初始各自一个标签
  → 迭代（最多10轮）：每个实体按邻居加权投票更新自己的标签
       投票权重 = 语义相似度(向量余弦)*0.6 + 关系连接*0.4
  → 收敛后合并相似社区（平均向量余弦 > 0.85）
  → LLM 为每个社区生成名称+摘要（没配模型则成员名拼接兜底）
```

- **LPA（Label Propagation Algorithm）**：经典社区发现算法，思想是「物以类聚」——每个节点不断采用邻居中最多的标签，迭代至收敛。优点是快、无需预设社区数。
- **双触发**：萃取写图后增量触发（新实体按邻居投票归入社区）+ Celery beat 每天凌晨全量兜底。

## 七、为什么用 Neo4j 而不是关系库

- 记忆天然是「实体-关系」的图结构；
- 高频操作是**多跳关系遍历**（取实体的一跳/多跳邻居），图数据库专门为此优化，关系库要多次自连接，复杂且慢；
- Neo4j 5 原生支持向量索引 + 全文索引，一个库满足图遍历 + 语义检索。

## 八、面试问答（八股）

**Q1：知识图谱是什么？三元组指什么？**
知识图谱是用「实体-关系-实体」描述知识的图结构。三元组 = (主语, 谓词, 宾语)，如（用户, 偏好, 周杰伦）。本项目从对话萃取三元组写入 Neo4j 构建用户画像图谱。

**Q2：怎么从文本萃取三元组？**
用 LLM。先把文本切成原子陈述句并做指代消解，再让 LLM 按受控词表（限定实体类型和谓词）抽出实体和三元组，用 jinja2 prompt 给定示例和 JSON 输出格式约束。受控词表防止实体类型泛滥。

**Q3：实体去重（消歧）怎么做？**
两层：批内去重（本次萃取的实体之间）+ 图侧融合（和图里已有实体）。策略是「规则初筛 + LLM 精判」：同名同类型直接合并（快路径），相似的（文本/向量相似度超阈值）才交 LLM 判定，置信度够才合并。避免全量问 LLM 的开销。

**Q4：为什么用图数据库存记忆？**
记忆是实体关系结构，本质是图。高频的多跳关系遍历用图库天然高效，关系库要多次自连接。Neo4j 还原生支持向量+全文索引，检索一体化。

**Q5：记忆检索和 RAG 检索有什么不同？**
RAG 检索非结构化文档块，记忆检索结构化实体图。记忆检索多了「一跳邻居关系遍历」——召回实体后把关联事实顺着关系边带出来，上下文更结构化。融合策略类似（向量+全文加权）。

**Q6：画像和事件为什么要区分？**
画像是稳定属性（「在腾讯工作」），事件是一次性有时间的经历（「3月看了演唱会」）。事件带 event_time 能排时间线，画像不需要。萃取时让 LLM 判断陈述是稳定事实还是一次性经历，分别落 Entity 和 Event 节点。

**Q7：社区聚类用什么算法？为什么？**
标签传播 LPA。每个节点迭代采用邻居中得票最高的标签直到收敛。优点：快、线性复杂度、无需预设社区数。本项目投票权重结合了语义相似度（向量余弦）和关系连接，比纯结构化 LPA 更贴合语义。

**Q8：MERGE 幂等是什么意思？**
Neo4j 的 MERGE「存在则匹配，不存在则创建」。配合节点 id 唯一约束，同一份数据重复写入只会更新属性、不会产生重复节点，保证萃取任务可安全重试。

**Q9：萃取是同步还是异步？为什么？**
异步（Celery）。萃取要多次调 LLM（陈述抽取+三元组+去重判定），很慢。主动记住/对话回答时只落库+派发任务立即返回，萃取在 worker 后台跑，完成回写状态。

**Q10：怎么保证萃取质量？**
①受控词表约束输出；②jinja2 prompt 给充分示例和严格 JSON schema；③指代消解（「我」统一为「用户」），不可解析的指代整条跳过；④健壮的 JSON 解析（容错 LLM 输出的格式瑕疵）；⑤萃取失败不中断流水线，单句失败跳过。

**Q11：向量索引在 Neo4j 里怎么用的？**
Neo4j 5 支持 `CREATE VECTOR INDEX`，指定维度和相似度函数（cosine）。实体的 name 向量化存 name_embedding 属性，检索用 `db.index.vector.queryNodes` 做 kNN。和 ES 向量检索同理，只是在图库里，召回后能直接接关系遍历。

## 九、常见追问（进阶）

**Q1：原子陈述（Statement）这一层为什么需要？直接从原文抽三元组不行吗？**
直接从长文本抽三元组，LLM 容易漏抽、把多个事实揉一起、指代混乱。先切成「原子陈述句」（一句话一个事实）并标注类型/时间、做指代消解，再逐句抽三元组，召回更全、更准、可溯源到具体哪句话。这是「分而治之」。

**Q2：陈述的类型（FACT/OPINION/PREDICTION/SUGGESTION）和时间类型有什么用？**
区分事实和观点/预测，便于后续按可信度处理；时间类型（STATIC 静态属性 / DYNAMIC 会变 / ATEMPORAL 无关时间）帮助判断这条记忆是否会过时。本项目主要用它辅助判断是否落成事件，企业级可做记忆更新/失效。

**Q3：实体去重时 LLM 判定的成本怎么控制？**
不全量两两问。先用廉价的规则初筛（文本相似度 difflib + 向量余弦 + 包含关系）筛出可疑候选对，只对超过阈值（0.8）的对才调 LLM 判定。同名同类型直接合并不问 LLM。把 LLM 调用量从 O(n²) 降到很少。

**Q4：两个实体名字一样但其实不是一个人怎么办（消歧）？**
靠 type + description + 别名 + 上下文。dedup prompt 让 LLM 综合判断（「张三」vs「张三的公司」类型不同不合并）。拿不准倾向不合并（宁可重复也别错合并污染图谱）。这是 entity resolution / disambiguation 问题。

**Q5：记忆会不会过时/冲突？怎么处理？**
本项目明确「记忆不遗忘」（砍掉了遗忘引擎）。关系边带 valid_at/invalid_at 时间属性预留了时效表达。冲突（「在腾讯」后来「在字节」）当前是新增不覆盖，靠 dialog_at 时间区分。企业级可做时序推理/失效，本项目按需裁剪了。

**Q6：LPA 聚类不稳定（每次结果不同）怎么办？**
标准 LPA 因更新顺序随机确实可能不稳定。本项目用加权投票（语义相似度+关系）降低随机性，加最大迭代次数和社区合并（相似社区合并）收敛。增量更新只动新实体不全量重算，保持稳定。要更稳可换 Leiden/Louvain 算法。

**Q7：四层溯源会不会让图特别大、查询变慢？**
节点数确实多（Dialogue/Chunk/Statement/Entity）。但检索主要在 Entity 层（向量+全文索引），溯源层只在需要回溯时遍历。个人数据量小没问题；大规模可对溯源层做归档/分离。索引（id 唯一约束 + user_id 属性索引）保证查询性能。

**Q8：为什么实体名向量化而不是整句向量化？**
记忆检索是「找相关实体」，用实体名（+描述）向量化，query 匹配的是实体本身。整句向量化是 RAG 文档块的做法。两者目标不同：RAG 找相关文本段，记忆找相关实体再带出关系。

**Q9：增量聚类和全量聚类怎么配合？**
萃取写图后增量触发：新实体按邻居投票归入已有社区或新建，快、只动局部。Celery beat 每天凌晨全量重聚类兜底：修正增量累积的偏差、合并该合并的社区。增量保实时、全量保质量。

**Q10：知识图谱可视化前端怎么做的？性能问题？**
用 AntV X6，后端返回全量 nodes/edges，前端力导向布局渲染，点击高亮邻居。个人数据量节点不多直接全量渲染。大规模要做：默认只渲染核心子图、按社区聚合、分页/懒加载、虚拟化。

**Q11：MENTIONS、RELATION、INVOLVES 三种边的区别？**
MENTIONS：Statement→Entity，溯源边（这句话提到了这个实体）。RELATION：Entity→Entity，语义三元组（实体间真实关系，带 predicate）。INVOLVES：Event→Entity，事件参与（这个事件涉及这些实体）。前者是溯源结构，后两者是知识语义。

**Q12：记忆分层（短期/长期）怎么实现？为什么不做遗忘？**
每个实体/陈述带 `memory_layer`（默认 short_term）。定时巩固任务按「access_count / importance / mention_count 任一达标」把短期提升为 long_term，**只升不降**。本项目明确「记忆不遗忘」——个人记忆助手的核心价值是长期留存用户信息，遗忘曲线（ACT-R/艾宾浩斯）适合工作记忆/缓存淘汰场景，与产品定位相悖，故刻意砍掉。分层的意义不是淘汰旧记忆，而是让稳定记忆在检索时优先、并触发画像增强。

**Q13：节点的 importance/access_count 这些属性怎么参与排序？会不会形成正反馈？**
检索融合分 = 0.55·向量 + 0.30·全文 + 0.15·importance，长期记忆额外 +0.05；命中后回写 access_count。确实是正反馈：越重要/越常被检索 → 排序越靠前 → 越容易再被命中 → 越快被巩固为长期。这正是我们想要的「高价值记忆浮现」，但要防马太效应淹没新记忆，所以新记忆仍能靠向量/全文相关度直接召回（importance 只占 0.15 权重，不主导），且巩固是「只升」不挤占新记忆的召回机会。
