# 情绪记忆与情绪化音乐推荐 — 设计与面试八股

> 结合 Comet 情绪记忆系统与情绪化音乐推荐的真实实现，讲解 情绪计算 / valence-arousal 模型 / 多信号融合推荐 / 外部音源接入 / 异步处理 等面试点。
> 对应代码：`api/app/core/emotion/`（ontology/analyzer/aggregator）+ `api/app/core/music/`（migu_client/mood_tagger/recommender）+ `api/app/services/{emotion_service,music_service}.py` + `api/app/tasks/{emotion,music}.py` + 前端 `web/src/components/MusicPlayer.tsx`、`web/src/pages/MusicLibraryPage.tsx`。

---

## 一、模块在做什么

两件事，前者为后者供数：

1. **情绪记忆**：每轮对话后异步分析用户当前情绪，结构化存储为「离散主情绪 + 效价 valence + 唤醒度 arousal」，滚动维护用户「当前情绪画像」。
2. **情绪化音乐推荐**：综合「记忆图谱里用户喜欢的歌手 + 当前对话情绪」自动选一首歌，配一台带旋转唱片与滚动歌词的悬浮播放器播放。

一句话：让助手不仅「记得你说过什么」，还「懂你此刻的心情」，并用一首恰到好处的歌回应。

---

## 二、情绪模型：valence-arousal 二维情感空间

业界常用的情感维度模型（Russell 情绪环）：任何情绪都可以投影到二维平面：

- **valence（效价）**：-1 极消极 ~ +1 极积极（难过↔愉悦）。
- **arousal（唤醒度）**：0 极平静 ~ 1 极激动（放松↔亢奋）。

例：喜悦≈(0.8, 0.7)、悲伤≈(-0.7, 0.3)、愤怒≈(-0.6, 0.8)、平静≈(0.2, 0.2)。

`core/emotion/ontology.py` 维护一张受控词表：13 类离散主情绪 + 每类的参考 (valence, arousal) 坐标。作用有三：①给 LLM 抽取时的锚定参考；②LLM 未给维度值时按主情绪兜底填充；③校验 LLM 给的维度是否与主情绪大致一致（偏差过大以词表为准，防幻觉）。

**为什么用二维连续坐标而不是只存离散情绪**：离散标签（开心/难过）粒度粗、无法计算「距离」；而歌曲推荐要做「情绪匹配」，需要把「当前情绪」和「歌曲情绪」放进同一连续空间算距离。**情绪和歌曲共用同一套 valence-arousal 坐标系，是两个模块能对接的关键**。

## 三、情绪分析与画像（写入侧）

### 1. 单轮情绪分析（`emotion/analyzer.py`，Celery 异步）

每轮对话回答后，fire-and-forget 派发情绪分析任务（不阻塞主回答）：

```
用户发言文本
  → LLM 结构化分析（jinja2 prompt + few-shot）
       输出：emotion_type / intensity / valence / arousal / keywords / trigger / summary
  → 规范化：主情绪入受控词表、valence/arousal clamp 到合法区间、缺失按词表兜底
  → 强度低于阈值（弱情绪）丢弃，不入库
  → 写 emotion_records + 刷新 emotion_profile
```

健壮性（对标记忆萃取）：LLM 调用有限重试（默认 2 次，退避）、`json_repair` 健壮解析、连续失败返回中性兜底 (0, 0.3) 不抛异常。

### 2. 情绪画像滚动聚合（`emotion/aggregator.py`）

`emotion_profiles` 表每用户一条，每次新记录后用**最近 N 条**滚动平均刷新：

- `avg_valence / avg_arousal`：最近 N 条算术平均；
- `dominant_emotion`：最近 N 条出现次数最多的主情绪（并列取累计强度更高者）；
- 中性记录也纳入统计，反映真实基线。

读侧 `GET /emotion/current` 返回 `{dominant_emotion, avg_valence, avg_arousal, sample_count, health_index}`，其中 health_index 由 avg_valence 线性映射到 0~100 作「情绪健康指数」。这个接口就是音乐推荐的情绪输入。

> 为什么异步：情绪分析要调 LLM，慢；对话回答是主流程不能等。和记忆萃取同款 fire-and-forget，任务级独立 DB 引擎（NullPool）+ asyncio.run。

## 四、音乐推荐（消费侧）

### 1. 选歌：多信号融合打分（`music/recommender.py`）

输入「曲库（每首已标情绪坐标）+ 目标情绪坐标 + 偏好歌手集合」，纯打分不调 LLM：

```
情绪匹配度 = 1 - 归一化(valence/arousal 欧氏距离)   # 越近越高，0~1
综合分 = 0.7·情绪匹配度 + (歌手命中用户偏好 ? +0.3 : 0)
按综合分降序排序
```

- **偏好歌手**从哪来：`search_memory("喜欢的歌手 音乐 歌曲")` 检索记忆图谱，抽出人物/音乐人类型实体或与音乐相关的实体名集合 —— 这就是记忆系统直接喂给推荐的数据。
- **情绪坐标**从哪来：`/emotion/current` 的 (avg_valence, avg_arousal)，无数据用中性 (0, 0.3)。
- **推荐队列**：取打分 top 段，对队首若干随机打散组成一条队列（既保证情绪相关、又让每次「为我推荐」换队首/换批，避免老推同一首），仅纳入「可播放」的歌。

### 2. 三层音源降级（核心工程取舍）

选出歌（歌名+歌手）后，音频按优先级取，任一层拿到即用：

1. **自建曲库（主，最稳）**：用户上传的本地 mp3，存对象存储，走鉴权文件接口播放——能放周杰伦等版权歌。
2. **咪咕免费歌（补充）**：调咪咕 `listenSong.do`，免费歌返回 302 重定向到真实 mp3 直链（带签名、有时效）；VIP 付费歌返回 200 无 location，自动落到下一层。
3. **仅展示（兜底）**：拿不到音频时只显示封面+歌名+「暂无音源」提示，不出声但有推荐。

> **为什么放弃 VIP 在线播**：VIP 歌取播放地址需要逆向客户端加密 + 登录态，不可靠且灰色；版权歌靠用户自己上传本地 mp3。这是务实的取舍。
>
> **为什么直链「现播现取」不入库**：咪咕免费直链带签名、会过期，存库会播不了；改为播放前实时解析（`GET /music/songs/{id}/audio`），不持久化。

### 3. 情绪标签 LLM 自动标注（`music/mood_tagger.py`）

用户上传只填歌名/歌手，valence/arousal/mood_tags 全由 LLM 推断（复用 emotion 的 valence-arousal 词表做锚点，few-shot 用知名华语歌：晴天清新怀旧、夜曲阴郁、孤勇者激昂）。失败兜底中性坐标 (0, 0.3)，不阻断上传。

### 4. 异步处理（`tasks/music.py`，v0.0.2 体验关键）

添加歌曲若同步做「补封面/歌词 + LLM 情绪标注 + 音源验证」要十几秒。改为：

```
添加歌曲 → 立即入库（playable=待验证, tag_status=pending）→ 派发 Celery 任务 → 秒返回
后台任务：补封面/歌词/专辑 + LLM 情绪标注 + 音源可用性验证 → 回写 playable/tag_status
前端：检测到 pending 每 3s 轮询刷新，处理完自动更新卡片状态
```

`playable` 字段（True/False/None）落库，推荐只选 `playable=True` 的歌，从源头保证「不推荐播不了的歌」。

## 五、前端：沉浸式播放器

- **悬浮播放器**（`MusicPlayer.tsx`）常驻右下角，迷你/展开两态。展开后：旋转的 CD 唱片（CSS `animation`，外圈 conic 渐变流光环）、随 `<audio>` 的 currentTime 解析 LRC 并居中高亮+自动滚动的歌词、进度条、推荐语、上一首/下一首。
- **播放队列**：推荐返回的是队列，切歌在队列内进行，且**切到无音源的歌会自动跳过、现取下一首能播的**。
- **本地音频鉴权加载**：`/api/files/..` 需带 token，`<audio>` 无法自带请求头，故前端先 `fetch` 带 token 取成 blob、生成 object URL 再喂给 audio。
- **主题跟随路由**：进入 `/music` 整站切深色霓虹沉浸主题，播放器在音乐页深色、其他页浅色变体融入界面。
- **曲库页**（`MusicLibraryPage.tsx`）深色玻璃「专辑墙」，卡片描边/封面光晕按该曲情绪坐标动态着色。

## 六、面试问答（八股）

**Q1：valence-arousal 是什么？为什么用它？**
情感计算里的二维情绪模型（Russell 环）：valence 效价（消极↔积极）、arousal 唤醒度（平静↔激动）。用连续坐标而非离散标签，是因为推荐要算「情绪距离」，离散标签无法度量远近。情绪和歌曲共用同一坐标系才能匹配。

**Q2：怎么分析用户情绪？实时吗？**
每轮对话后异步（Celery）调 LLM 结构化分析，输出主情绪+valence+arousal+关键词等，规范化后写库并滚动刷新画像。异步是因为调 LLM 慢、不能阻塞对话主流程；弱情绪按强度阈值丢弃避免噪声。

**Q3：当前情绪画像怎么算的？**
最近 N 条情绪记录滚动平均：valence/arousal 取算术平均，主导情绪取出现次数最多（并列取累计强度高者）。滚动窗口让画像反映「最近一段」而非全历史。

**Q4：音乐推荐的输入有哪些？怎么融合？**
两路信号：①记忆图谱检索出的偏好歌手；②情绪系统的当前 valence/arousal。打分 = 0.7·情绪匹配度(1-归一化欧氏距离) + 0.3·歌手命中。这是典型的多信号加权融合推荐。

**Q5：为什么不直接用 LLM 选歌？**
选歌是「在已标坐标的曲库里按距离打分」，纯数值计算又快又可控又免费；LLM 适合做「情绪标注」「推荐语生成」这类语言任务。MVP 推荐语也用模板，不为每次推荐调 LLM，省 token、低延迟。

**Q6：版权音乐怎么解决的？三层降级讲一下。**
本地上传 mp3（最稳，能放版权歌）→ 咪咕免费歌在线（listenSong 302 取真实直链，VIP 拿不到自动降级）→ 仅展示卡片。VIP 在线播要逆向加密+登录态，不可靠故放弃，版权歌靠用户上传。

**Q7：咪咕直链为什么不存数据库？**
免费直链带签名、有时效，存库后会过期播不了。改为播放前实时解析（现播现取），不持久化。曲库只存 file_key（本地）或元数据，播放时按「本地>外链>咪咕现取」解析真实地址。

**Q8：添加歌曲为什么要异步？怎么做的？**
补封面/歌词、LLM 情绪标注、音源验证加起来十几秒，同步会让用户干等。改为入库即返回（pending），派发 Celery 任务后台处理，回写 playable/tag_status，前端轮询刷新。和文档解析/记忆萃取同一套异步范式。

**Q9：怎么保证不推荐播不了的歌？**
歌曲后台处理时验证音源可用性，落 `playable` 字段（True/False/None）。推荐只在 playable=True 的歌里打分选择；播放器切歌时若遇到无音源也会自动跳过、现取下一首能播的。双重保证。

**Q10：歌词滚动高亮怎么实现的？**
解析 LRC（带 [mm:ss.xx] 时间戳）成按时间排序的行数组，监听 `<audio>` 的 timeupdate，按 currentTime 二分/线性定位当前行，高亮并 scrollIntoView 居中。容器要 position:relative 才能正确按 offsetTop 居中。

**Q11：本地音频带鉴权，HTML5 audio 怎么加载？**
`<audio src>` 不能自定义请求头，没法直接带 token 访问受保护的 `/api/files/..`。方案：前端用 fetch 带 Authorization 取文件成 blob，`URL.createObjectURL` 生成临时本地地址再喂给 audio，组件卸载时 revoke 释放。图片鉴权加载同理。

**Q12：情绪系统和记忆系统、音乐推荐是怎么串起来的？**
情绪系统产出「当前情绪画像」(/emotion/current)，记忆系统产出「偏好歌手」(search_memory)，音乐推荐同时消费这两路 + 自建曲库的情绪坐标做融合打分。三者通过统一的 valence-arousal 坐标系和记忆检索接口解耦协作——情绪和记忆都是「上游能力」，音乐推荐是第一个同时消费两者的「体验型功能」。
